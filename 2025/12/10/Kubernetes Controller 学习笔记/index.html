Kubernetes Controller 学习笔记

一、控制器基础概念

什么是控制器

控制器（又称工作负载 Workload）是Kubernetes中的中间层管理工具，通过标签选择器（label-selector）与Pod关联，实现对Pod的全生命周期管理。

核心职责：

  • 创建和删除Pod
  • 监听Pod运行状态
  • 故障自动重建
  • 支持扩容缩容
  • 支持版本升级和回滚

控制器与Pod的关系

  • 控制器 通过 label-selector 与 Pod 关联
  • Pod 通过控制器实现运维管理（伸缩、升级等）

二、常见控制器类型

控制器 用途 使用占比 核心特性
Deployment 无状态应用 60%~70% 支持滚动更新、回滚、声明式管理
StatefulSet 有状态应用 10%~20% 有序部署、稳定网络标识、专属存储
ReplicaSet 副本集管理 - 维护Pod副本数（通常由Deployment管理）
DaemonSet 守护进程 特殊场景 每个节点运行一个Pod
Job 一次性任务 测试、批处理 任务完成后退出（complete状态）
CronJob 周期性任务 巡检、监控 按时间表周期执行
Ingress L7层代理 - 负载均衡、流量路由（将被Gateway API替代）

四、StatefulSet（有状态应用控制器）

与Deployment的根本差异

Deployment 适合无状态应用（如Web服务），Pod可互换，副本无序。

StatefulSet 适合有状态应用（如数据库、中间件），每个Pod有唯一身份和专属存储。

三大核心特性

1. 稳定的网络标识

  • 基于 Headless Service（无头服务，ClusterIP为None）⭐
    pod_ip:port访问服务，headless不需要使用kube-proxy做内外网映射
  • Pod名称格式固定：<Pod名称>.<Service名称>.<命名空间>.svc.cluster.local
  • 例如：myapp-0.myapp-svc.default.svc.cluster.local

2. 稳定的持久化存储

  • 通过 volumeClaimTemplates 为每个Pod自动创建PVC
  • 每个Pod拥有独立的存储卷，数据不随Pod删除而丢失⭐
  • Pod重新调度后仍能访问相同数据

3. 有序部署与伸缩

  • 扩容：按顺序创建（0 → 1 → 2）
  • 缩容：按逆序删除（2 → 1 → 0）
  • 确保前一个Pod就绪后才操作下一个

StatefulSet组成（三部分）

组件 作用
Headless Service 提供Pod的DNS解析，无ClusterIP
volumeClaimTemplates 为每个Pod生成PVC，绑定专用PV
StatefulSet 管理Pod副本，保证网络标识和存储稳定性

五、DaemonSet（守护进程控制器）

核心特性

  • 自动部署：新节点加入集群时自动创建Pod
  • 自动清理：节点移除时自动删除相关Pod
  • 每节点一个：每个节点（或符合条件的节点）运行一个Pod副本

适用场景

  • 存储守护进程：glusterd、ceph
  • 日志收集：fluentd、logstash
  • 监控代理：Prometheus Node Exporter、Datadog

六、Job（一次性任务控制器）

核心特性

  • 一次性执行：任务完成后容器成功退出，不再重启
  • 重试机制：失败时可配置重试
  • 批处理：适合离线、批量数据处理

适用场景

  • 数据库迁移
  • 批处理脚本
  • 离线数据处理
  • 视频解码

九、控制器对比总结

Deployment vs StatefulSet

特性 Deployment StatefulSet
应用类型 无状态应用 有状态应用
Pod名称 随机，无固定标识 固定且有序（0,1,2…）
DNS 无固定DNS记录 稳定的DNS记录
存储 共享存储 每个Pod独立存储
部署顺序 无序 有序部署（0→N-1）
缩容顺序 无序 有序缩容（N-1→0）
Service ClusterIP Headless Service

Deployment vs DaemonSet

特性 Deployment DaemonSet
副本分布 根据副本数随机分配 每个节点运行一个
适用场景 无状态应用 守护进程（日志、监控）
扩缩容 通过replicas控制 自动与节点数同步
节点感知 不感知节点变化 感知节点加入/移除

Job vs CronJob

特性 Job CronJob
执行方式 一次性执行 周期性执行
应用场景 批处理、数据处理 定时备份、巡检
时间计划 无 需要schedule
重试机制 backoffLimit控制 并发策略控制

十、关键概念梳理

Pod更新策略（重要）

  • 滚动升级：逐步替换Pod，保证服务连续可用（Deployment默认）
  • 蓝绿升级：一次性替换所有Pod，有风险但快速
  • 金丝雀升级：灰度发布，先小比例替换，验证后再全量
  • 敏捷发布：基于特性开关的灰度方案

标签与选择器

  • 标签（Label）：贴在资源上的键值对，用于分组和选择
  • 选择器（Selector）：根据标签条件筛选资源
  • 控制器通过选择器与Pod关联：controller.matchLabels = pod.labels

存储相关

  • PV（PersistentVolume）：集群级别的存储资源
  • PVC（PersistentVolumeClaim）：Pod对存储的申请
  • StatefulSet的volumeClaimTemplates：为每个Pod自动创建PVC

Headless Service

  • ClusterIP: None：无虚拟IP
  • 用途：直接解析到Pod IP，用于有状态应用服务发现
  • DNS格式：<Pod名>.<Service名>.<namespace>.svc.cluster.local

十一、实践建议

集群部署前检查

常见问题处理

  1. StatefulSet删除后数据丢失：需要手动删除PVC
  2. PV状态为Released：删除PV再重新创建，或修改回收策略
  3. Pod日志权限问题：绑定cluster-admin权限给system:anonymous

云平台对应关系

  • 服务器 → ECS
  • 负载均衡 → SLB
  • 镜像仓库 → ACR
  • Kubernetes → ACK
  • 数据库 → RDS
  • 日志分析 → SLS
  • 数据库迁移 → DTS
