Kubernetes 网络、RBAC 和 HPA 相关知识

一、Kubernetes 网络体系详解

1.1 三层网络架构

Kubernetes 集群网络由三个独立的网络层组成，每层承担不同的责任：

节点网络（Node Network）

  • 定义：负责集群宿主机节点间的通信，并打通与集群外部的连接
  • IP 类型：nodeIP
  • 实现基础：物理网卡或云网络基础设施
  • 通信方式：通过物理网卡上的 nodeIP 实现跨主机通信
  • 典型场景：集群内部节点相互访问、外部客户端访问集群入口

Pod 网络（Pod Network）

  • 定义：为集群内所有 Pod 提供网络通信能力
  • IP 类型：podIP，全局唯一
  • 关键特性：Pod 可以跨节点相互通信，无论它们运行在同一节点还是不同节点
  • 实现机制：通过 CNI（Container Network Interface）网络插件实现
  • 网络模型：
    • Overlay（覆盖网络）：数据包经过二次封装，增加额外处理但支持跨网段
    • Route（路由网络）：基于路由表转发，性能更优但对网络基础设施有要求

Service 网络（Service Network）

  • 定义：集群部署时指定的虚拟网络，Service 从中分配 IP
  • IP 类型：clusterIP，虚拟 IP（仅在集群内有效）
  • 实现方式：Service 的 IP 实际存在于 ipvs 或 iptables 规则中，由 K8S 自行管理
  • 通信方式：通过 Service 的 clusterIP 实现对后端 Pod 的代理转发和负载均衡
  • 特性：具有内置的服务发现和负载均衡能力

1.2 CNI 插件详解

CNI 全称：Container Network Interface（容器网络接口）

核心职责：

  • 为每个 Pod 分配全局唯一的 podIP
  • 实现 Pod 跨主机通信
  • 管理容器网络的生命周期

管理者：kubelet 负责调用 CNI 插件完成网络配置

Flannel - 简单易用方案

适用场景：中小规模集群（节点数 ≤ 10 个）

默认网段：10.244.0.0/16

三种工作模式：

UDP 模式（最早实现）
  • 性能：最差（应用层处理封装）
  • 跨网段：支持
  • 配置：最简单
  • 适用：学习和小规模测试
host-GW 模式（性能最优）
  • 性能：最好
  • 跨网段：不支持（需要二层可达）
  • 配置：复杂
  • 原理：每个节点的 Pod 网段直接配置为目标节点的下一跳
  • 适用：高性能要求、网络环境可控的场景
VXLAN 模式（默认推荐）
  • 性能：折中方案（内核处理，性能优于 UDP）
  • 跨网段：支持
  • 配置：中等难度
  • 优势：在内核态处理，减少上下文切换

Flannel 的核心思想：侧重于数据包的封装与解封装，通过数据包本身的改造实现跨节点通信

Calico - 功能丰富方案

适用场景：中大规模集群（节点数 > 10 个）

默认网段：192.168.0.0/16

核心特性：

  • 支持复杂的网络策略（NetworkPolicy）
  • 内置数据加密和安全隔离
  • 支持流量整形和 QoS
  • 功能完善但配置复杂

四种工作模式：

IPIP 模式
  • 性能：较好
  • 原理：将整个 IP 包作为数据负载，再封装到另一个 IP 包中
VXLAN 模式
  • 与 Flannel 的 VXLAN 类似
  • 性能介于 IPIP 和 BGP 之间
BGP 模式（性能最优，推荐）通信效率高——只通过路由规则完成交互

一旦创建新的Pod，felix会创建节点对应的路由信息，然后通知给bird，bird会告知给邻接节点上的bird，然后给与felix 学习到这个Pod 所在的节点路由路径，从而完成跨节点转发
简而言之，BGP模式，把所有pod 视为一个独立的”节点”，使用动态路由策略，来完成路径寻址，以及转发。

核心组件：

  • Felix：Calico Agent，运行在每个节点上，维护路由规则
  • BIRD：BGP 客户端，负责路由分发和学习

工作流程：

数据转发过程：

  • 性能：最优（无封包解包开销，直接路由转发）
  • 跨网段：支持（BGP 可跨越网络边界）
  • 配置：复杂（需要理解 BGP）
  • 可扩展性：极强（支持大规模集群）

BGP 模式的核心思想：将每个 Pod 视为独立的”路由节点”，利用动态路由协议（BGP）完成跨节点寻址和转发

混合模式（cross-subnet）
  • 在同网段采用 BGP 高效转发
  • 跨网段时自动切换到 VXLAN 封装

Calico 的核心思想：侧重于策略角度实现跨节点通讯，以及提供细粒度的网络控制

Cilium - 新一代方案

  • 融合 Flannel 和 Calico 的优点
  • 基于 eBPF 技术实现
  • 性能更优，功能更强
  • 学习成本较高

1.3 CNI 选型对比详解

特性 Flannel Calico
模式 UDP / host-GW / VXLAN IPIP / VXLAN / BGP / 混合
默认网段 10.244.0.0/16 192.168.0.0/16
传输方式 叠加网络（Overlay），IP 隧道 路由转发（BGP 最优），或叠加网络
性能排序 host-GW > VXLAN > UDP BGP > VXLAN > IPIP
网络策略支持 无 支持丰富的 NetworkPolicy
跨网段 Flannel 支持 Calico BGP 不直接支持，混合模式支持
配置难度 简单（开箱即用） 复杂（需定制优化）
资源消耗 低 中高
适用规模 小规模（< 10 节点） 大规模（> 10 节点）
附加功能 无 数据加密、安全隔离、流量整形、QoS

1.4 生产环境选型建议

小规模集群（5-10 节点）：

  • 推荐：Flannel + host-GW（如果网络可控）或 Flannel + VXLAN
  • 优势：部署简单、资源占用少、运维负担低

中等规模集群（11-50 节点）：

  • 推荐：Calico + BGP 或混合模式
  • 优势：性能稳定、支持网络策略、易于扩展

大规模集群（50+ 节点）：

  • 推荐：Calico + BGP 或 Cilium
  • 优势：性能优异、功能完善、支持细粒度控制

2.2 RBAC 四大核心资源对象

主体（Subject）- “谁”要权限

K8S 中有三种主体：

1. ServiceAccount（SA）- Pod 身份

  • 用于 Pod 访问 API Server
  • 每个 Namespace 自动创建 default SA
  • Pod 自动挂载 SA 的 Token、CA 证书和 Namespace 到 /var/run/secrets/kubernetes.io/serviceaccount/
  • Pod 与 API Server 交互时使用 SA 的 Token 进行身份验证

2. User - 用户

  • 通常代表真实的人或系统用户
  • 通过证书或 Token 认证
  • 格式：字符串，系统前缀为 system: 保留

3. Group - 用户组

  • 将多个 User 组织在一起
  • 便于权限批量分配
  • 格式与 User 相同

权限定义（Role 和 ClusterRole）- “有什么”权限

Role - 命名空间级权限

资源操作权限说明：

  • get：获取单个资源
  • list：列举资源
  • watch：监听资源变化
  • create：创建资源
  • update：更新资源
  • patch：部分更新资源
  • delete：删除资源
  • exec：在 Pod 中执行命令
  • attach/port-forward：调试操作

ClusterRole - 集群级权限

关键差异：

  • Role 权限仅在其 Namespace 内生效
  • ClusterRole 权限在整个集群范围内生效

权限绑定（RoleBinding 和 ClusterRoleBinding）- 权限”生效范围”

RoleBinding - 命名空间级绑定

将 Role 或 ClusterRole 绑定到主体，权限仅在该 Namespace 生效：

ClusterRoleBinding - 集群级绑定

将 ClusterRole 绑定到主体，权限在整个集群范围内生效：

重要特性：

  • RoleBinding 可以引用 Role 或 ClusterRole
  • ClusterRoleBinding 只能引用 ClusterRole
  • RoleBinding 中引用 ClusterRole 时，权限仍限于该 Namespace

2.6 RBAC 最佳实践

  1. 最小权限原则

    • 只授予必要的权限
    • 定期审计和清理过期权限

  2. 使用 ServiceAccount 而非 User

    • Pod 访问 API Server 必须使用 SA
    • 便于权限版本管理

  3. 合理区分 Role 和 ClusterRole

    • 命名空间隔离的权限用 Role
    • 跨命名空间或系统级权限用 ClusterRole

  4. 使用 Group 管理权限

    • 批量分配权限更高效
    • 便于部门或团队权限管理

  5. 定期审计权限

三、HPA（Horizontal Pod Autoscaling）详解

3.1 HPA 核心概念

定义：Horizontal Pod Autoscaling，Pod 的水平自动扩缩容机制

核心作用：根据应用的资源使用情况自动调整 Pod 副本数，实现弹性伸缩

与传统扩容的区别：

  • 手动扩容：需要人工干预，响应慢
  • HPA 自动扩容：根据指标自动调整，实时响应负载变化

3.2 HPA 工作原理

3.3 前置条件：metrics-server

必须部署 metrics-server，否则 HPA 无法工作

作用：

  • 收集集群中所有节点和 Pod 的资源使用情况
  • 为 HPA 提供 CPU、内存等指标数据
  • 支持 kubectl top 命令

部署验证：

如果看不到指标：

  • 检查 metrics-server 是否已启动：kubectl get deployment metrics-server -n kube-system
  • 检查 metrics-server 日志：kubectl logs -n kube-system deployment/metrics-server
  • 等待 metrics 收集完成（通常 1-2 分钟）

3.4 HPA 核心参数详解

基础命令创建

参数说明

--cpu-percent（目标 CPU 利用率）

  • 单位：百分比（0-100）
  • 含义：当前 CPU 平均利用率超过此值时触发扩容
  • 例如：--cpu-percent=50 表示 CPU 利用率 > 50% 时扩容
  • 计算逻辑：
    • 当前 CPU 利用率 = (当前 CPU 用量) / (Pod 的 CPU Request) × 100%
    • 目标副本数 = ceil(当前副本数 × 当前利用率 / 目标利用率)

--min（最小副本数）

  • 默认值：1
  • 含义：即使无负载也至少保留的 Pod 数
  • 用途：保证服务可用性

--max（最大副本数）

  • 默认值：10
  • 含义：最多可以扩容到的 Pod 数
  • 用途：防止资源耗尽和成本失控

其他重要参数（YAML 定义时）：

参数详解：

  • stabilizationWindowSeconds：稳定窗口

    • 扩容（scaleUp）：默认 0 秒，立即响应
    • 缩容（scaleDown）：默认 300 秒（5 分钟），防止频繁波动

  • Percent / Pods：扩缩容方式

    • Percent：按百分比扩缩（相对增减）
    • Pods：按固定数量扩缩（绝对增减）

五、面试高频题目

5.1 网络相关

Q1：Flannel 和 Calico 的区别是什么？

A：

  • Flannel：简单易用，侧重封包解包，适合小集群（< 10 节点）
  • Calico：功能丰富，支持网络策略，适合大集群（> 10 节点）
  • 性能：Calico BGP 模式无封包开销，性能最优
  • 选型：小规模用 Flannel，大规模用 Calico

Q2：BGP 模式如何工作的？

A：

  • Felix 为每个 Pod 创建路由规则
  • BIRD 使用 BGP 协议分发路由给邻接节点
  • 每个节点学习其他节点的路由
  • 数据直接通过路由转发，无需封装

Q3：Pod 跨节点通信的完整流程？

A：

  1. 源 Pod 发送数据包到目标 Pod
  2. 根据目标 podIP 查找路由规则
  3. 根据规则转发到目标节点的物理网卡
  4. 目标节点收包后，再转发到目标 Pod
  5. （如果使用 Overlay 网络，还要加上封包解包）

5.2 RBAC 相关

Q1：RBAC 的四个核心对象是什么？

A：

  • Subject：User、Group、ServiceAccount（谁）
  • Role/ClusterRole：权限定义（什么）
  • RoleBinding/ClusterRoleBinding：权限绑定（在哪生效）

Q2：Role 和 ClusterRole 的区别？

A：

  • Role：命名空间级，权限限于该 Namespace
  • ClusterRole：集群级，权限跨越所有 Namespace

Q4：Pod 如何访问 API Server？

A：

  • SA 自动挂载 Token、CA 证书到 /var/run/secrets/…
  • Pod 使用 Token 与 API Server 交互

5.3 HPA 相关

Q1：HPA 的工作原理？

A：

  • 每 15 秒检查一次指标
  • 计算目标副本数 = 当前副本数 × 当前利用率 / 目标利用率
  • 如果目标副本数 > 当前副本数，则扩容
  • 如果目标副本数 < 当前副本数，则缩容（5 分钟冷却）

Q2：HPA 无法工作的原因？

A：检查清单：

  • metrics-server 是否部署？
  • Pod 是否配置了 CPU request？
  • 指标是否收集成功？（kubectl top pods）
  • 等待时间是否足够？（15-30 秒）

Q3：HPA 与 VPA 的区别？

A：

  • HPA：水平扩缩（增删 Pod），适合无状态服务
  • VPA：垂直扩缩（调整资源），适合单个 Pod 资源不足

Q4：如何触发 HPA 扩容？

A：

  1. 部署应用并配置 resources.requests
  2. 创建 HPA 对象并设置 cpu-percent
  3. 向应用发送大量请求增加负载
  4. 监控 HPA 状态，Pod 数量逐渐增加

五、常见面试题

Q1：K8s 有几道安全防线？ A：三道。认证（谁）→ 鉴权（权限）→ 准入控制（策略）

Q2：Pod 如何访问 API Server？ A：通过 Service Account。Pod 自动挂载 SA token、CA 证书，通过 token 认证和后续的 RBAC 鉴权。

Q3：Role 和 ClusterRole 的区别？ A：Role 作用于某个命名空间，ClusterRole 作用于全集群。RoleBinding 可以绑定 ClusterRole，但仍限于该 namespace。

Q4：权限是加法还是减法？ A：权限是白名单加法模式。即使拥有高权限角色，也不会自动获得下级权限；权限只能累加。

Q5：kubeconfig 里有什么？ A：三部分。集群信息（CA+API Server 地址）、用户认证信息（证书）、上下文（用户→集群→namespace）。

Q6：什么情况下使用 Service Account？ A：Pod 内的应用需要访问 API Server 时。例如 dashboard、coredns、监控 agent 等。

Q7：RoleBinding 中 CN 和 O 字段分别指什么？ A：CN（Common Name）是用户名，O（Organization）是用户组名。API Server 从客户端证书解析这两个字段确认身份。

K8S CNI、RBAC、HPA 常见面试题

一、CNI 插件（Container Network Interface）

基础概念

CNI 是什么？

  • CNI = 容器网络接口标准
  • K8s 的网络实现不是内置的，而是通过 CNI 插件来完成
  • 作用：为 Pod 分配 IP、配置网络、实现 Pod 间网络通讯

面试题精选

Q1：K8s 中网络通讯分几层？分别用什么实现？

A：三层（从外到内）

各层职责：

  • Ingress：基于域名/路径做反向代理，暴露到集群外
  • Service：服务发现，通过 endpoints 找到后端 Pod
  • CNI 插件：为 Pod 分配 IP，实现 Pod 到 Pod 的网络通讯
  • Pause 容器：同 Pod 内的容器共享网络命名空间，通过 localhost 通讯

Q2：为什么需要 CNI 插件？K8s 为什么不自己实现？

A：CNI 是标准，K8s 需要支持不同的网络实现

原因：

  1. 网络是多样化的
    • 云环境（AWS/阿里云）有自己的网络模型
    • 私有数据中心网络结构不同
    • 不同企业有不同的网络策略
  2. K8s 设计哲学：核心功能 + 可插拔扩展
    • CRI（容器运行时）可以是 Docker/Containerd
    • CNI（网络）可以是 flannel/calico/weave
    • CSI（存储）可以是各厂商存储驱动

类比：就像手机系统，不能内置所有驱动，而是提供标准接口让厂商实现

Q3：常见的 CNI 插件有哪些？区别是什么？

A：常见的有 5 种

插件 网络模型 性能 网络策略 适用场景
flannel Overlay（vxlan） 中等 ❌ 不支持 小集群、开发测试
calico 纯三层（BGP） 高 ✅ 支持 大集群、生产推荐
weave Overlay 中等 ✅ 支持 混合环境
cilium eBPF 最高 ✅ 支持 高性能场景
AWS/阿里云 云厂商网络 高 依赖厂商 公有云专用

一般都用vxlan模式

核心区别：

Flannel（被吐槽的选择）

Calico（生产推荐）

简单介绍其中三个

关键差异：

  • Flannel 是 overlay（虚拟网络），Calico 是纯三层（真实路由）
  • Calico 支持 NetworkPolicy，flannel 不支持
  • Calico 性能比 flannel 好
