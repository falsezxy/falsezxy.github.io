Kubernetes 存储管理

一、ConfigMap 与 Secret 配置管理

1.1 ConfigMap（配置管理）

概念

ConfigMap 用于管理 Pod 中容器内服务的配置文件和环境变量。

适用场景

  • 存储应用配置文件（如 Nginx.conf、数据库连接配置）
  • 定义环境变量
  • 非加密的配置数据

典型例子：Ruo-Yi 微服务

  • 前端：Nginx + VUE 框架（DIST 文件）
  • 配置文件：Nginx.conf

1.2 Secret（加密管理）

概念

Secret 管理 Pod 中容器的加密信息、令牌与密钥文件。

适用场景

  • 存储加密密钥
  • 管理用户凭证（用户名/密码）
  • 存储 API Token 和 Secret 密钥文件
  • 敏感信息保护

Secret 创建方式

方式一：使用 kubectl 命令创建

方式二：Base64 编码后编写 YAML

Secret 使用方式

方式一：直接挂载（Volume Mount）

特点：将加密文件/配置文件直接挂载到容器中使用

方式二：环境变量注入

特点：通过环境变量方式为容器提供账号与密码的对应关系

ConfigMap vs Secret 对比

特性 ConfigMap Secret
用途 管理非敏感配置信息 管理敏感加密信息
数据类型 明文配置、环境变量 Base64 编码的敏感数据
使用场景 应用配置文件、环境配置 凭证、密钥、Token、API密钥
编码方式 无编码 Base64 编码

二、Kubernetes 存储卷详解

2.1 存储卷的必要性

问题背景

  1. 容器生命周期短暂：容器崩溃时数据丢失，重启后恢复到镜像初始状态
  2. 多容器文件共享：Pod 内多个容器需要共享文件数据
  3. 数据持久化需求：应用数据需要长期保存

解决方案

Kubernetes 通过 Volume 抽象 和 Pause 容器 的共享命名空间实现数据持久化和共享。

2.2 存储卷分类

第一类：emptyDir（临时存储卷）

特点：

  • Pod 分配到节点时创建，Pod 删除时数据永久丢失
  • 存储在节点的临时目录中，初始为空
  • Pod 中所有容器可读写同一 emptyDir

实现原理：

  • 基于 Pause 容器的共享挂载/存储命名空间
  • 支持边车模式（Sidecar Pattern）：主容器 + 副容器共享存储

适用场景：

  • 临时数据存储（缓存、日志临时文件）
  • 多容器间文件共享（如配置初始化、数据预处理）

配置示例：

第二类：hostPath（节点存储卷）

特点：

  • 将 Node 节点上的文件或目录挂载到 Pod 中
  • 实现单节点持久化，但数据绑定到特定 Node
  • Pod 转移到其他节点后，无法访问之前的数据

工作原理：

  • Pod 直接使用所在 Node 节点的挂载目录
  • 数据绑定到单个节点，跨节点不可用

适用场景：

  • 单节点数据持久化（特定节点的应用配置）
  • 调试场景（访问节点系统文件）
  • 监控类应用（收集节点指标）

配置示例：

局限性：

  • Pod 删除后数据保留，但转移节点后无法使用
  • 不适合多副本部署场景
  • 数据可靠性依赖节点健康状况

第三类：NFS（网络共享存储）

特点：

  • 将 NFS 服务器的共享目录挂载到 Pod
  • 支持跨节点访问，任何 Node 上的 Pod 都可访问
  • 数据存储在 NFS 服务器，高可靠性

工作原理：

  • 将 Pod 环境视为 NFS 客户端
  • Pod 直接 NFS 挂载，访问 NFS 服务器共享目录

适用场景：

  • 多节点共享数据
  • 需要持久化存储的应用（数据库、文件存储）
  • 多 Pod 副本需要访问同一数据

  • 数据持久化在 NFS 服务器

第四类：PV/PVC + StorageClass（企业级持久化存储）

核心概念：

组件 说明
PV（PersistentVolume） 集群中的存储资源抽象，由运维人员创建和管理，独立于 Pod 生命周期
PVC（PersistentVolumeClaim） 用户对存储资源的请求，描述所需的存储特性（容量、访问模式等）
StorageClass 存储类模板，动态创建 PV，自动化管理存储资源

静态 PV 与动态 PV 对比

步骤 静态创建 PV 动态创建 PV
1. 准备存储设备 确保存储设备可用并配置好 确保存储设备可用并配置好
2. 创建 PV 资源 手动创建 PV，定义访问模式、存储空间、回收策略 无需手动创建 PV，由 StorageClass 自动创建
3. 创建 PVC 资源 手动创建 PVC，绑定已创建的 PV 创建 PVC，指定 StorageClass，动态生成并绑定 PV
4. 创建 Pod 资源 挂载 PVC 存储卷 挂载 PVC 存储卷
5. 存储卷插件 无需额外配置 需配置 Provisioner（如 NFS-Client）
6. RBAC 授权 无需 RBAC 授权 需创建 ServiceAccount 和 RBAC 授权
7. StorageClass 无需 StorageClass 需创建 StorageClass 关联 Provisioner
适用场景 存储需求固定，PV 数量较少 存储需求多样，PV 数量较多
运维复杂度 较高，需手动管理 PV 和 PVC 较低，自动化管理
灵活性 灵活性较低，需提前规划 灵活性高，按需动态创建

选择建议

使用 emptyDir：

  • 临时缓存、日志等非持久数据
  • Pod 内多容器数据共享（边车模式）

使用 hostPath：

  • 特定节点的监控应用
  • 单节点工具类应用
  • 开发调试环境

使用 NFS：

  • 中小规模多节点共享需求
  • 无需复杂存储管理的场景
  • 快速原型化和测试

使用 PV/PVC：

  • ⭐ 生产环境推荐
  • 有状态应用（数据库、消息队列等）
  • 需要灵活存储管理的场景
  • 企业级持久化存储需求

三、有状态应用与服务管理

3.1 StatefulSet（有状态应用控制器）

概念： 专为有状态应用设计的 Kubernetes 工作负载，保证 Pod 的稳定性、唯一性和顺序性

适用场景：

  • 数据库（MySQL、MongoDB、Redis）
  • 消息队列（Kafka、RabbitMQ）
  • 集群应用（Elasticsearch、Cassandra）

3.2 Service 与 Headless Service

Service 普通服务

  • 提供稳定的虚拟 IP 和 DNS 名称
  • 通过 kube-proxy 进行流量转发

Headless Service（无头服务）

特点：

  • 无虚拟 IP，不需要 kube-proxy 映射
  • 通过 CoreDNS 直接解析 Pod 名称为 Pod IP
  • 支持直接访问特定 Pod：pod-name.service-name.namespace.svc.cluster.local

优势：

  • 降低网络延迟
  • 支持 Pod 间直接通信
  • 适合有状态应用的 Pod 间协调

四、运维工程师日常工作

核心职责

  1. 存储管理： PV/PVC、StorageClass 的创建和维护
  2. 配置管理： ConfigMap、Secret 的部署和更新
  3. 监控维护： 存储卷的健康检查和数据备份
  4. 故障处理： PV 绑定问题、存储容量告警、数据恢复

面试常见问题

Q1: ConfigMap 和 Secret 的区别？
A: ConfigMap 存储明文配置信息，Secret 存储加密敏感数据。使用场景不同：ConfigMap 用于应用配置，Secret 用于凭证和密钥。

Q2: PV/PVC 与 NFS 有什么区别？
A: NFS 是存储类型，PV/PVC 是 K8S 的存储资源抽象。NFS 可以直接挂载到 Pod，PV/PVC 提供更灵活的存储管理和多存储后端支持。

Q3: 什么时候使用静态 PV，什么时候使用动态 PV？
A: 存储需求固定且数量少→静态 PV；存储需求多样且频繁变化→动态 PV + StorageClass。

Q4: emptyDir 数据会丢失吗？
A: 是的。emptyDir 中的数据只要 Pod 在节点上运行就会保存，Pod 删除后数据永久丢失。不适合需要持久化的场景。

五、最佳实践

存储选择决策树

安全管理原则

  1. Secret 加密： 始终使用 Base64 编码敏感信息
  2. RBAC 权限： 限制 ConfigMap/Secret 访问权限
  3. 备份策略： 重要数据定期备份到 NFS/云存储
  4. 监控告警： 监控存储容量，设置告警阈值

生产环境建议

  • ✅ 必用 PV/PVC：提供统一的存储管理和故障隔离
  • ✅ StorageClass 自动化：避免人工错误，提高效率
  • ✅ 多副本部署：确保高可用性和数据安全
  • ✅ 定期备份：重要数据备份到多个位置
  • ✅ 性能优化：选择合适的存储后端（SSD/HDD）
