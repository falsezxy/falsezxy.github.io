Kubernetes Pod 学习笔记

Pod 基础概念

Pod 是 Kubernetes 中最小的调度与运行单位，是对一个或多个容器的封装。Kubernetes 直接管理的是 Pod，而不是容器。

Pod 的核心特性

  1. 共享资源

    • 同一个 Pod 中的容器运行在同一个 Node 节点上
    • 容器之间通过 localhost 互相通信
    • 共享同一个 IP 地址和端口范围
    • 共享同一个存储卷（Volume），实现容器间的数据共享

  2. 生命周期

    • Pod 内的容器共享生命周期
    • 如果 Pod 被删除，容器也随之消失
    • Pod 是临时性的，可能被删除或重建

Pod 的三种容器类型

图例：

1️⃣ Pause 容器（基础架构容器）

定义：Pod 中第一个被启动的容器，也称为 Infrastructure Container、Sandbox Container

核心作用：

  • 持有网络命名空间：为 Pod 分配 IP 地址，所有业务容器通过加入 Pause 容器的网络命名空间来共享这个 IP
  • 生命周期管理：充当 Pod 的”父容器”，如果 Pause 容器挂了，整个 Pod 的网络环境就消失了
  • 僵尸进程回收：充当 PID 1 进程，回收 Pod 内部可能产生的僵尸进程

特点：

  • 非常小（几百 KB），几乎不占用资源
  • 用户不可见，由 K8s 自动管理
  • 与 Pod 共存亡

形象比喻：Pause 容器就是房子的地基、水电管道和门牌号。虽然你不住在地基里，但如果没有它，房子就不复存在，水电（网络）也通不了。

2️⃣ Init 容器（初始化容器）

定义：在业务容器启动之前运行的容器，一个 Pod 可以有一个或多个

核心作用：

  • 顺序执行：Init 容器是串行运行的，必须等第一个成功完成（Exit Code 0），第二个才开始。只有所有 Init 容器都成功退出，业务容器才会启动
  • 包含特殊工具：可以在 Init 容器中安装 curl、dig、netcat 或数据库迁移脚本，使业务容器镜像保持精简和安全
  • 阻塞等待依赖：可以写脚本循环检查依赖服务（如 MySQL）是否就绪，只有依赖服务就绪才启动业务容器，避免启动报错

使用场景：

  • 初始化配置文件
  • 检查和等待依赖服务
  • 数据库迁移
  • 环境预处理

形象比喻：Init 容器就像”装修队”。在业主（业务容器）搬进来之前，装修队先进场，负责把墙刷好、家具装好、确认水电通畅。只有装修队干完活，业主才能搬进来。

示例：

3️⃣ 业务容器（App Containers）

定义：运行用户应用程序的容器，如 Spring Boot、Nginx、Python 脚本等

核心作用：

  • 长期运行：负责持续运行和提供服务
  • 并行启动：一旦 Init 容器全部完成，所有的业务容器会同时（并行）启动
  • 共享环境：共享 Pause 容器创建的网络、IPC 和 UTS，可以挂载相同的存储卷共享文件

Sidecar 模式：

  • 有时一个 Pod 里有两个业务容器：主应用（Main App）和辅助应用（Sidecar）
  • Sidecar 与主应用同生共死，如日志收集、流量代理等
  • 形象比喻：主容器是主人（赚钱养家），Sidecar 是管家或保姆（辅助主人）

形象比喻：业务容器就是住在房子里的”业主”，负责赚钱养家（处理核心业务流量）。

容器类型对比表

特性 Pause 容器 Init 容器 App 容器
启动顺序 第 1 个 第 2 阶段（串行） 第 3 阶段（并行）
生命周期 与 Pod 共存亡 运行完即退出 长期运行
核心职责 提供网络命名空间、IP、挂载点 准备环境、检查依赖、初始化 处理核心业务逻辑
资源消耗 极低（几百 KB） 仅在启动时消耗 视应用负载而定
用户感知 不可见，自动管理 可选，用户定义 核心，用户定义

Pod 的三种类型

1. 自主式 Pod

特点：

  • 由 Scheduler 调度到 Node 节点部署
  • 不被控制器管理，没有自愈能力（Pod 挂了不会被重建）
  • 没有副本管理能力，不支持滚动更新
  • 配置信息存储在 etcd 中

2. 控制器管理的 Pod

特点：

  • 由 Scheduler 调度到 Node 节点部署
  • 被控制器（如 Deployment、StatefulSet）管理
  • 具有自愈能力（Pod 挂了会被重建）
  • 支持副本管理和滚动更新
  • 配置信息存储在 etcd 中

Pod 容器的镜像拉取策略

通过 imagePullPolicy 字段定义，决定容器启动时如何获取镜像。

三种策略

策略 说明 默认适用于
IfNotPresent 优先使用 Node 本地镜像，本地没有才从仓库拉取 非 latest 标签的镜像
Always 总是从仓库拉取，无论本地是否存在 latest 标签或无标签的镜像
Never 仅使用 Node 本地镜像，总是不从仓库拉取 离线场景

Pod 容器的重启策略

通过 restartPolicy 字段定义，决定容器退出时的行为。

三种策略

策略 说明 默认适用于
Always 当容器退出时，总是重启容器（无论退出状态码如何） Deployment、StatefulSet、DaemonSet 等控制器管理的 Pod
OnFailure 仅当容器异常退出时（状态码非 0），重启容器；正常退出（状态码 0）不重启 Job 等任务型 Pod
Never 当容器退出时，总是不重启容器 需要手动清理的 Pod

重要提示：Deployment、StatefulSet、DaemonSet 控制器的 Pod 容器重启策略只能设置为 Always

Pod 容器的资源限制

通过 resources 字段定义容器可使用的资源。

资源配置项

CPU 和内存单位

CPU 单位：

  • m（毫核）：1000m = 1 核
  • 浮点数：0.5 = 500m

内存单位：

  • 2 进制：Ki、Mi、Gi（1Gi = 2^30 字节）
  • 10 进制：k、M、G（1G = 1,000,000,000 字节）
  • 默认单位为字节（B）

资源限制规则

  • 若容器 CPU 使用超过 limits.cpu，会被限流，但容器不会被终止
  • 若容器内存使用超过 limits.memory，会触发 OOM（内存不足）错误，进程被终止并 Pod 重启
  • requests 用于调度器选择节点，limits 用于运行时限制

QoS 服务质量等级

QoS（Quality of Service）等级决定了 Pod 的调度优先级和节点资源紧张时的驱逐顺序。

三个等级

QoS 等级 条件 优先级 驱逐时机
Guaranteed 所有容器均设置 requests 和 limits，且二者相等 最高 资源不超限时不会被驱逐
Burstable 至少一个容器设置了 requests 或 limits（或两者部分设置） 中 资源紧张且超过 requests 时可能被驱逐
BestEffort 所有容器均未设置 requests 或 limits 最低 资源不足时优先被驱逐

Pod 容器的健康检查（探针）

通过探针定义来检测容器状态，根据检查结果决定是否重启或调整服务流量。

三种探针类型

探针类型 作用 触发场景
livenessProbe 检测容器是否正常运行 失败则重启容器（根据重启策略）
readinessProbe 检测容器是否就绪（可接收流量） 失败则从 Service 端点列表中移除，不再转发流量
startupProbe 检测应用是否启动成功 成功前暂停 liveness 和 readiness 探针，防止启动期间被误判为失败

三种探测方式

方式 说明 适用场景
exec 执行命令，返回码为 0 则成功 脚本或命令检查
httpGet 发送 HTTP 请求，状态码 2XX/3XX 为成功 Web 服务健康检查
tcpSocket 建立 TCP 连接，端口可达则成功 TCP 服务健康检查

Pod 容器的生命周期钩子

通过 lifecycle 配置容器启动和终止时的操作。

两种钩子

钩子 触发时机 用途 特点
postStart 容器创建后、容器进程启动之前 初始化任务、资源部署、配置设置 执行失败不会导致容器重启
preStop 容器即将被 Kubelet 终止时 优雅关闭、保存状态、释放资源 与 SIGTERM 信号异步执行，若超时会被强制终止

postStart 钩子

触发时机：容器创建后、容器进程启动之后（即容器主进程开始运行但不一定完全准备好接收请求时）

特点：

  • 即使钩子执行失败，容器仍会被启动，Kubelet 不会因此重启容器
  • 可以执行初始化任务，如资源部署、环境配置

preStop 钩子

触发时机：当容器即将被 Kubelet 终止时

特点：

  • 与 SIGTERM 信号异步执行
  • 如果钩子执行时间超过优雅终止期（grace period，默认 30 秒），Kubernetes 将强制终止 Pod
  • 常用于优雅关闭应用，关闭连接、保存状态、清理资源

示例：完整的生命周期钩子配置

执行流程：

  1. Init 容器执行，向 /var/log/nginx/message 写入 “Hello initContainers”
  2. 主容器启动后，postStart 钩子执行，追加 “Hello from postStart”
  3. 容器运行…
  4. Pod 被删除时，preStop 钩子执行，追加 “Hello from preStop”
  5. SIGTERM 信号发送，容器优雅终止

Pod 生命周期状态详解

Pod 从创建到销毁会经历多种状态。

Pod 的五个主要状态

状态 说明
Pending Pod 已创建但容器还未启动，通常在等待资源或镜像下载
Running 容器已启动并正在运行
Succeeded Pod 中的所有容器都已成功退出（通常用于 Job 类型的 Pod）
Failed Pod 中至少有一个容器异常退出
Unknown 无法获取 Pod 的状态

Pod 常见的详细状态

状态 原因 解决方案
ContainerCreating 容器正在被创建 等待即可
ImagePullBackOff 镜像拉取失败，正在重试 检查镜像名称、仓库地址、网络、认证信息
ErrImagePull 镜像拉取出错 检查镜像是否存在、仓库地址是否正确
CrashLoopBackOff 容器启动后立即崩溃，陷入重启循环 查看日志 kubectl logs pod-name，检查应用代码或配置
OOMKilled 容器内存溢出被杀死 增加 resources.limits.memory
Terminating Pod 正在被优雅终止 等待或强制删除 kubectl delete pod --force --grace-period=0
Error 容器运行时出错 查看详细信息 kubectl describe pod pod-name

常见面试题

基础概念类

Q1: Pod 是什么？为什么 K8s 不直接管理容器？

A: Pod 是 Kubernetes 最小的调度单位，可以包含一个或多个容器。K8s 不直接管理容器的原因：

  1. 提供网络抽象：Pod 内容器共享 IP，简化网络管理
  2. 提供存储抽象：Pod 内容器可共享存储卷
  3. 资源管理：统一对 Pod 进行资源限制和分配
  4. 容器隔离：Pod 提供逻辑隔离，多个 Pod 可独立运行

Q2: Pod 中有哪几种容器，各自的作用是什么？

A:

  1. Pause 容器（基础架构容器）：

    • 第一个启动的容器
    • 为 Pod 分配 IP 地址
    • 所有业务容器加入 Pause 容器的网络命名空间共享 IP
    • 充当 PID 1 进程回收僵尸进程

  2. Init 容器（初始化容器）：

    • 在业务容器启动前执行
    • 串行运行，必须都成功才启动业务容器
    • 用于环境准备、依赖检查、数据库迁移等
    • 保持业务容器镜像精简和安全

  3. 业务容器（App 容器）：

    • 长期运行，提供实际服务
    • 并行启动（除非使用 SidecarContainer 特性）
    • 包括主应用和 Sidecar 辅助容器

Q3: Pod 的三种类型是什么，生产环境应该用哪种？

A:

  1. 自主式 Pod：不被控制器管理，没有自愈能力，不推荐
  2. 控制器管理的 Pod：被 Deployment、StatefulSet 等管理，具有自愈能力，生产环境首选
  3. 静态 Pod：由 kubelet 直接管理，通常用于集群组件，不推荐用户应用

生产环境必须使用控制器管理的 Pod，以获得自动恢复、副本管理、滚动更新等能力。

镜像和重启策略类

Q4: imagePullPolicy 有哪几种，默认是什么？

A: 三种策略：

  1. IfNotPresent：优先使用本地镜像，本地没有才从仓库拉取（默认，适用于非 latest 标签）
  2. Always：总是从仓库拉取（默认，适用于 latest 标签或无标签）
  3. Never：只使用本地镜像，不从仓库拉取

Q5: restartPolicy 有哪几种，Deployment 的 Pod 只能设置哪种？

A: 三种策略：

  1. Always：总是重启（默认，适用于 Deployment、StatefulSet、DaemonSet）
  2. OnFailure：仅异常退出时重启（适用于 Job）
  3. Never：不重启（适用于需要手动清理的 Pod）

Deployment 的 Pod 只能设置 Always，这是 Kubernetes 的强制约束。

资源管理类

Q6: requests 和 limits 的区别是什么？

A:

  • requests：容器启动时需预留的资源（保证值），用于调度器选择节点
  • limits：容器可使用的最大资源（上限值），运行时限制，超过会被限流或 OOM

两者的关系：

  • requests <= limits
  • CPU 超限：限流（降速）
  • 内存超限：OOM 错误，进程被杀死

Q7: 什么是 QoS，有哪三个等级，驱逐顺序是什么？

A: QoS（Quality of Service）决定 Pod 的优先级和驱逐顺序：

  1. Guaranteed：requests == limits，优先级最高，不会被驱逐
  2. Burstable：至少设置了 requests 或 limits，优先级中等，资源不足可能被驱逐
  3. BestEffort：未设置 requests 和 limits，优先级最低，资源不足优先被驱逐

驱逐顺序：BestEffort > Burstable > Guaranteed

健康检查类

Q9: liveness、readiness 和 startup 三种探针的区别是什么？

A:

探针 作用 失败后果
livenessProbe 检测容器是否正常运行 重启容器
readinessProbe 检测容器是否就绪（可接收流量） 从 Service 端点移除，不转发流量
startupProbe 检测应用是否启动成功 暂停其他探针，防止误判

使用场景：

  • liveness：确保容器持续健康运行
  • readiness：确保流量只转发给就绪的 Pod
  • startup：等待启动时间较长的应用完全启动

Q10: 探针有哪三种检测方式，各自的优缺点是什么？

A:

  1. exec：执行命令，返回码 0 成功

    • 优点：灵活，可执行复杂脚本
    • 缺点：开销大，需要命令行工具

  2. httpGet：发送 HTTP 请求，2XX/3XX 成功

    • 优点：标准方式，开销小
    • 缺点：仅适用于 Web 服务

  3. tcpSocket：建立 TCP 连接，可达成功

    • 优点：开销小，快速
    • 缺点：无法检查应用逻辑，只能检查端口

选择建议：

  • Web 服务优先 httpGet
  • TCP 服务使用 tcpSocket
  • 复杂逻辑使用 exec

生命周期类

Q11: postStart 和 preStop 的区别是什么，各用在什么场景？

A:

  • postStart：容器启动后执行，用于初始化任务、资源部署、配置设置；失败不会导致容器重启
  • preStop：容器终止前执行，用于优雅关闭、保存状态、释放资源；与 SIGTERM 异步执行

使用场景：

  • postStart：预热缓存、初始化配置、部署资源
  • preStop：关闭数据库连接、保存状态、清理临时文件、优雅关闭应用

Q12: Pod 的生命周期状态有哪些，Terminating 状态多久会强制杀死？

A: 五个主要状态：Pending、Running、Succeeded、Failed、Unknown

常见详细状态：

  • Pending：等待资源或镜像
  • ContainerCreating：容器创建中
  • Running：运行中
  • ImagePullBackOff：镜像拉取失败重试
  • CrashLoopBackOff：容器启动失败重试循环
  • Terminating：正在优雅终止

Terminating 时间：

  • 默认优雅终止期为 30 秒（terminationGracePeriodSeconds: 30）
  • 30 秒后会被强制杀死
  • 可通过 kubectl delete pod --grace-period=60 自定义

故障排查类

Q13: 排查 Pod 故障的三步法是什么？

A:

  1. kubectl get pods → 查看 Pod 状态，识别异常
  2. kubectl describe pod → 查看详细信息和 Events，定位问题原因
  3. kubectl logs → 查看日志，分析具体错误

逐步深化排查，最终根据事件和日志分析并解决问题。

Q14: CrashLoopBackOff 状态表示什么，怎么排查？

A: CrashLoopBackOff 表示容器启动后立即崩溃，陷入重启循环。

排查步骤：

常见原因：

  • 应用代码有 bug
  • 配置文件错误
  • 依赖服务未就绪（应该使用 Init 容器等待）
  • 容器启动命令错误
  • 权限问题

Q15: ImagePullBackOff 状态怎么排查？

A: ImagePullBackOff 表示镜像拉取失败，K8s 正在重试。

排查步骤：

常见原因：

  • 镜像标签不存在
  • 私有仓库地址错误
  • 私有仓库认证信息缺失或错误
  • 网络不可达

进阶题

Q17: Pod 中容器共享的资源有哪些，不共享的有哪些？

A:
共享的：

  • 网络命名空间（IP、端口）
  • IPC 命名空间（进程间通信）
  • UTS 命名空间（主机名）
  • 存储卷（Volume）

不共享的：

  • 进程命名空间（PID，但可以配置 shareProcessNamespace）
  • 文件系统（除了挂载卷）
  • 用户命名空间
