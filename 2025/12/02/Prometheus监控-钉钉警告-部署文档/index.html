Prometheus监控+钉钉警告 部署指南

一、简介及作用

简介

Prometheus 是一个开源的监控和告警系统，广泛应用于云原生环境中。它通过拉取模式主动采集目标系统的时间序列指标数据，经过灵活的告警规则处理后，由 AlertManager 统一管理和分发告警。钉钉作为企业级实时通讯工具，可通过自定义机器人接收告警消息，让运维团队在第一时间获知系统异常，大大提高了问题的发现和响应效率。

本指南将指导你整合 Prometheus + AlertManager + 钉钉，构建一套企业级的监控告警系统。

系统作用

实时监控

  • 全面采集服务器、应用、数据库等各类系统指标
  • 支持自定义指标和应用埋点
  • 毫秒级的数据采集和存储

智能告警

  • 基于阈值和规则的灵活告警机制
  • 支持告警分组、去重和路由
  • 避免告警风暴，提高信号信噪比

及时通知

  • 与钉钉深度集成，告警实时推送到企业群
  • 支持 Markdown 格式的丰富告警消息
  • 构建完整的告警追踪链路

历史数据

  • 长期保存监控数据用于趋势分析
  • 支持数据查询和离线分析
  • 便于问题根因分析和性能优化

适用场景

  • 服务器监控：CPU、内存、磁盘、网络等资源告警
  • 应用监控：应用响应时间、错误率、吞吐量等性能指标
  • 数据库监控：数据库连接数、查询时间、慢查询告警
  • 业务监控：自定义业务指标的异常告警
  • 基础设施：容器集群、Kubernetes 节点健康状态
  • 依赖服务：缓存、消息队列、中间件等核心服务监控

二、工作流程和核心概念

工作流程

数据流向说明

  • Prometheus 定期抓取各 Exporter 的指标
  • 本地存储时间序列数据
  • 根据告警规则判断是否触发告警
  • 告警发送至 AlertManager 处理
  • AlertManager 通过 Webhook 调用钉钉接口
  • 告警信息显示在钉钉群中

三、核心概念

3.1 指标（Metrics）

Prometheus 采集的数据单位，包括：

  • Gauge：可增可减的指标（如温度、内存使用率）
  • Counter：只增不减的指标（如请求总数）
  • Histogram：观察值分布统计（如请求延迟）
  • Summary：分位数统计（如响应时间分位数）

3.2 告警规则（Alert Rules）

定义何时生成告警的条件，包括：

  • 规则名称和标签
  • 条件表达式（PromQL）
  • 持续时间（Duration）
  • 告警级别（Severity）

3.3 告警状态

  • Firing：告警条件满足，告警处于活跃状态
  • Pending：告警条件满足，但未达到持续时间
  • Resolved：告警条件不再满足，告警已解除

4.4 告警分组和路由

  • 分组：将相同类型的告警合并，避免消息风暴
  • 路由：根据告警标签将其分发到不同的接收器
  • 去重：避免相同告警的重复发送
  • 静音：临时禁止发送特定告警

四、部署指南

Alert部分部署流程简述：

在所有服务都部署完成的情况下，先添加对应告警的组件，配置相关api和接口，接着配置alertmanager的配置文件，添加告警组件部分的配置信息（邮件，钉钉等），重启alert服务，接着配置prometheus，自定义规则文件(比如/alert_rules/xxx.yaml)，配置prometheus.yml文件，添加alert_manager的target监听ip:端口，添加rules文件读取路径，重新加载prometheus完成告警配置。

五、常见问题及故障排查

5.1 钉钉机器人收不到告警消息

症状描述

  • AlertManager 已成功启动，告警规则已触发，但钉钉群中无消息显示

排查步骤

1.检查 prometheus-webhook-dingtalk 服务状态

确保服务正常运行且监听 8060 端口。

2.验证 Webhook URL 和 Secret 配置

确保 URL 和 Secret 与钉钉群机器人设置完全匹配，特别注意：

  • Webhook URL 中的 access_token 是否正确
  • Secret（加签）是否完整无误
  • 配置文件格式是否正确（YAML 缩进问题常见）

3.检查 AlertManager 配置文件

确认 webhook_configs 中的 URL 指向正确的 dingtalk 服务地址：

4.查看 AlertManager 日志

查找是否有发送失败的错误信息。

5.查看 dingtalk 服务日志

6.测试网络连接

5.2 Prometheus 告警规则不生效

症状描述

  • Prometheus 中虽然有告警规则，但没有触发任何告警

排查步骤

1.检查告警规则文件语法

2.验证 PromQL 表达式

  • 登录 Prometheus Web UI（http://192.168.118.101:9090）
  • 在 Graph 页面中测试告警规则中的 PromQL 表达式
  • 确保表达式能返回正确的数据

3.检查告警规则是否加载

4.验证数据采集

  • 在 Prometheus Web UI 的 Targets 页面确认目标状态为 UP
  • 确保 node_exporter 正常运行在目标主机上

5.检查告警持续时间

  • 确认条件满足的时长是否达到规则中的 for 定义
  • 例如规则设置 for: 1m，需要条件持续 1 分钟才会触发

5.3 Node Exporter 采集不到数据

症状描述

  • Prometheus Targets 页面显示某个 node_exporter 为 DOWN 状态
  • 或 Prometheus 无法获取到目标主机的指标

排查步骤

1.检查 node_exporter 服务状态

2.检查防火墙规则

3.测试网络连接

4.检查 Prometheus 配置中的 targets

确保目标 IP 地址和端口配置正确。

5.4 AlertManager 无法连接到 Prometheus

症状描述

  • AlertManager 启动正常，但无法从 Prometheus 接收告警

排查步骤

1.检查 Prometheus 配置

确保配置中 AlertManager 的地址和端口正确。

2.验证 AlertManager 服务

3.检查 Prometheus 日志

4.重新加载 Prometheus 配置

六、总结

本指南详细介绍了如何部署和配置一套完整的 Prometheus 监控 + 钉钉告警系统。通过本指南的学习和实践，你已经掌握了以下核心内容：

6.1 核心组件理解

  • Prometheus：作为监控数据收集和存储的中心，通过拉取模式采集各 Exporter 提供的指标
  • AlertManager：负责告警的聚合、去重和分发，提供灵活的路由机制
  • prometheus-webhook-dingtalk：将 AlertManager 的告警转换为钉钉消息的桥梁
  • Node Exporter：在被监控主机上采集系统级指标

6.2 部署关键步骤

  1. 部署 Prometheus Server 并配置数据保留策略
  2. 在被监控主机上部署 Node Exporter
  3. 配置 prometheus-webhook-dingtalk 并关联钉钉机器人
  4. 部署 AlertManager 并配置路由规则
  5. 编写告警规则文件并关联 AlertManager
  6. 测试告警流程确保端到端工作正常

6.3 生产环境建议

  • 高可用部署：在生产环境中应部署多个 AlertManager 实例，使用负载均衡
  • 数据备份：定期备份 Prometheus 的时间序列数据库
  • 告警分级：根据业务重要性设置不同的告警级别和接收器
  • 告警优化：合理设置告警分组和去重策略，避免告警风暴
  • 监控覆盖：逐步扩展监控范围，包括应用层、数据库、缓存等关键组件
  • 文档维护：保持告警规则文档和联系人信息的最新状态

6.4 后续学习方向

  • 深入学习 PromQL 查询语言，编写更复杂的告警规则
  • 探索 Grafana 与 Prometheus 的集成用于数据可视化
  • 了解 AlertManager 的分组、静音等高级特性
  • 学习如何集成其他通知渠道（邮件、Slack、PagerDuty 等）
  • 研究容器环境下的 Prometheus 监控实践
