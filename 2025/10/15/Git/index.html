设置Git的换行符处理

问题背景： 不同操作系统处理文本文件换行的方式不一样：

  • Windows：用两个字符换行（回车+换行，简称 CRLF）
  • Linux/macOS：只用一个字符换行（换行符，简称 LF）

当你和使用不同操作系统的人一起协作开发项目时，这种差异可能会导致文件格式混乱，Git 会误以为文件被修改了。

解决方法： 因为你用的是 Ubuntu（Linux 系统），运行这个命令： 如果是windows—则core.autocrlf true

这个命令的作用：

  • 当你提交代码时：自动把 CRLF 转换成 LF（统一格式）
  • 当你拉取代码时：保持原样不变（因为你的系统本来就用 LF）

这样可以保证代码仓库里的文件都使用统一的换行符格式（LF），避免因为换行符不同而产生的问题。

三、处理文件和暂存区

暂存区

为什么需要暂存区？

暂存区就像你的手提箱——你可以向其中添加物品（更改），如果你改变主意，可以删除物品，当你对手提箱里的所有东西感到满意时，你就把它拉上拉链（提交）。

通过git add命令告诉Git跟踪文件后，Git会将其添加到暂存区，包含到下一次的提交中

忽略特定文件.gitignore

有时，我们可能不希望 Git 跟踪某些文件，例如临时文件或敏感信息。Git 允许使用名为**.gitignore**的特殊文件忽略这些文件。

将问价存入**.gitignore**中即可让Git忽略其中的文件

可以看见输出中并没有列出debug.log

.gitignore文件非常强大。它可以忽略特定文件、整个目录，或使用模式匹配来忽略与特定条件匹配的文件。这在实际项目中非常有用，在这些项目中，可能有不应成为 Git 存储库一部分的构建工件、缓存文件或特定于环境的配置文件。

查看更改

取消暂存更改

如果我们在提交项目前改变主意，不想在下次提交中提交包含这些更改的文件，可以使用git restore命令来取消暂存文件

例如取消hello.py的暂存提交

小结

暂存区域，它是一项强大的功能，它允许我们通过仔细选择要包含的更改来制作更有意义的提交。暂存区重要的原因：

  1. 选择性提交：暂存区域允许我们仅提交部分更改，从而帮助我们创建更小、更有针对性的提交。
  2. 忽略文件：通过**.gitignore**排除不需要版本控制的文件来帮助保持存储库干净。
  3. 审查更改：git diff允许您在提交之前仔细检查更改，帮助您及早发现错误。
  4. 灵活性：取消暂存更改的能力使您可以自由地改变主意，从而使 Git 使用起来不那么令人生畏。

四、Git分支操作

分支的作用：
隔离开发 - 多人在不同分支独立工作，互不影响

特性开发 - 为新功能创建专门分支，完成后合并

bug 修复 - 快速创建修复分支，解决问题后合并

版本管理 - 用不同分支代表不同版本（如 main、develop）

代码审查 - 通过 Pull Request 审查代码后再合并

并行开发 - 多个分支同时进行，提高效率

实验空间 - 创建临时分支尝试，不满意可删除

删除不需要的分支

小结

掌握的高级 Git 操作：

  1. 修改提交：修复小错误或添加小改动，无需创建新提交，保持历史干净
  2. 恢复提交：创建新提交撤销更改，非破坏性方法，适合协作环境
  3. 挑选提交：精确选择特定提交应用到其他分支，灵活控制合并
  4. 交互式变基：重新排序、合并、编辑提交，在共享前清理本地历史
  5. 根变基：重塑整个项目历史，提供完全控制但风险极高

这些工具能维护清晰、有序的提交历史，提升代码审查效率、便于调试和理解项目演进。关键是：本地操作保持灵活，共享分支谨慎为上。

小结

掌握的核心概念：

  1. 存储更改：用 git stash 临时保存未完成的工作
  2. 应用更改：用 apply 保留存储，用 pop 删除存储
  3. 从存储创建分支：用 stash branch 将隐藏的改动转为新分支
  4. 管理多个存储：用消息标识，需要时查看和应用特定存储
  5. 清理存储：用 drop 删除单个或 clear 删除全部

使用要点

  • Stash 用于快速上下文切换，无需提交未完成的工作
  • 始终为存储添加描述性消息便于管理
  • 不要积累过多存储，长期工作改用分支或提交
  • Stash 是临时方案，不适合长期存放工作

软复位：移动头部

什么是软复位

软复位将 HEAD（和当前分支）移动到不同的提交，但不改变暂存区域或工作目录。适合需要撤销提交但保留所有更改的情况。

执行软复位

将 HEAD 移回两个提交。参数说明：

  • --soft ：软复位模式
  • HEAD~2 ：当前 HEAD 之前的两个提交（~N 表示向后 N 个提交）

查看复位结果

输出显示：

  • 前两个提交的所有更改已暂存
  • 工作目录文件未改变
  • 所有变更已准备好作为单个提交提交

将更改重新提交

将撤销的多个提交”压缩”为一个新提交。

使用场景

软复位特别有用于：

  • 压缩多个提交为一个
  • 修改多个提交的消息
  • 重新组织提交历史

⚠️ 重要警告

  • 软复位会重写历史记录
  • 如已推送原始提交到远程，需要强制推送（git push --force）
  • 强制推送可能给协作者带来问题
  • 在重写共享历史前，必须与团队沟通！

安全性

✅ 相对安全：所有更改都保留在暂存区或工作目录中，不会丢失数据

混合复位：取消暂存更改

什么是混合复位

混合复位是 git reset 的默认模式。它会：

  • 移动 HEAD
  • 更新暂存区域以匹配新的 HEAD
  • 不改变工作目录

创建示例场景

已修改文件并暂存更改。

执行混合复位

取消暂存更改，但保留在工作目录中。

查看结果

输出显示：

  • math.js 已修改但未暂存
  • 所有更改仍在工作目录中
  • 暂存区已清空

混合复位 vs 软复位

方面 软复位 混合复位
HEAD 移动 移动
暂存区 不改变 更新
工作目录 不改变 不改变
默认模式 需指定 默认

使用场景

混合复位特别有用于：

  • 取消暂存不想提交的更改
  • 想进一步审查或修改已暂存的内容
  • 需要重新整理暂存内容

安全性

✅ 很安全：

  • 所有工作都保留在工作目录中
  • 没有任何内容被丢弃
  • 可以随时重新暂存或修改

硬重置：放弃更改

什么是硬重置

硬重置是最剧烈的重置类型。它会：

  • 移动 HEAD
  • 更新暂存区域
  • 更新工作目录以匹配新 HEAD
  • 丢弃所有之后的更改

执行硬重置

此命令：

  1. 暂存并提交 math.js 的更改
  2. 执行硬重置到上一个提交
  3. 有效地”撤销”上一个提交并丢弃所有更改

查看结果

查看 math.js 文件，会发现除法函数已消失，就像从未写过一样。

三种重置模式对比

模式 HEAD 暂存区 工作目录 数据丢失
软复位 ✓ ✗ ✗ 否
混合复位 ✓ ✓ ✗ 否
硬重置 ✓ ✓ ✓ 是

⚠️ 使用硬重置的危险

  • 会永久丢弃所有更改
  • 无法轻易恢复丢弃的工作
  • 特别危险用于共享分支

使用场景

适合：

  • 完全放弃一些工作
  • 从特定提交重新开始
  • 清理意外提交

最佳实践

在硬重置前：

  1. ✅ 仔细检查重置目标提交
  2. ✅ 备份重要工作
  3. ✅ 考虑先创建新分支测试
  4. ✅ 如不确定，使用软或混合复位
  5. ✅ 避免在共享分支上使用硬重置

安全替代方案

如果担心数据丢失，可使用 git reflog 恢复（后续章节介绍）

使用 Reflog 恢复丢失的提交

什么是 Reflog

Reflog 是 HEAD 在本地存储库中所有位置的日志。它记录所有操作历史，包括：

  • 提交
  • 分支切换
  • 重置
  • 其他改变 HEAD 的操作

查看 Reflog

显示最近执行的所有操作列表(历史记录)，每个条目都有 HEAD@{n} 标识符。

Reflog 的优势

✅ 强大的安全网：

  • 可从几乎任何 Git 事故恢复
  • 记录所有 HEAD 移动
  • 捕获历史重写命令

⚠️ Reflog 的局限

  • 本地仓库专用，不同步到远程
  • 临时数据：条目通常保留 30-90 天后删除
  • 不能替代备份

最佳实践

不能完全依赖 Reflog：

  1. ✅ 定期备份重要工作
  2. ✅ 及时推送到远程仓库
  3. ✅ 使用分支隔离实验
  4. ✅ 在执行危险操作前创建备份分支

Reflog 是强大的安全工具，但不是绝对保险。

基于时间的复位

什么是基于时间的复位

使用时间戳而不是提交哈希将存储库重置到特定时间点的状态。适合记得大致变更时间但不知道具体提交的情况。

执行基于时间的复位

将存储库重置到 1 小时前的状态。

支持的时间描述符

Git 支持多种时间格式：

  • "1 hour ago" ：1 小时前
  • "yesterday" ：昨天
  • "2 days ago" ：2 天前
  • "3 minutes ago" ：3 分钟前
  • "1 week ago" ：1 周前
  • "2 months ago" ：2 个月前

使用场景

基于时间的复位特别适用于：

  • 记得大致何时出现问题
  • 想返回到某个工作日期
  • 不知道具体提交哈希

⚠️ 注意事项

  • 精确度较低：可能不如目标提交精确
  • 验证很重要：复位后务必检查仓库状态
  • 多个提交可能匹配：选择最接近指定时间的提交

验证复位结果

确保已到达预期的状态。

  2. ✅ 复位后立即验证结果
  3. ✅ 如不确定，先在新分支测试
  4. ✅ 保持 reflog 用于快速恢复
