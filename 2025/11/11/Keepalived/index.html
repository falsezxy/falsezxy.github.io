Keepalived是什么？

Keepalived是一个基于VRRP（Virtual Router Redundancy Protocol）协议的高可用解决方案。它可以在服务故障时自动进行故障转移，确保服务的连续性。

keepalived怎么实现高可用？

1、需要定义一个热备组(⭐)
2、确定主-备关系（通过优先级（100 - 90）来确定）
3、该怎么分辨MASTER能否正常工作呢？？
健康检查（icmp 协议—ping 检测存活）这是默认的检查方式
其他：例如 脚本检测服务的存活性
4、VIP的特性：
① VIP 是作为访问入口的IP地址（用于外网域名解析的 IP记录）
② VIP 只会存在于优先级最高的节点
5、怎么让VIP 漂移呢？
① keepalived 当发现master节点状态异常了，则会把master的优先级进行降级
例如：从100 -》80
以上机制建立完成后，Nginx 就可以与keepalived 组合实现HA 高可用了

Keepalived的核心功能

1. VRRP虚拟路由器功能

VRRP允许多个路由器（或服务器）共享一个虚拟IP地址（VIP），形成一个虚拟路由器组。

工作原理：

  • Master持有虚拟IP，优先级较高
  • Backup监听Master的心跳包
  • 如果Master故障，Backup自动接管虚拟IP

2. 故障检测与自动转移

Keepalived定期检测应用层服务的状态：

如果主服务故障，自动切换到备服务。

3. 抢占式与非抢占式

  • 抢占式：Master恢复后立即抢回VIP
  • 非抢占式：Master恢复后不抢回VIP，保持Backup为主

4. 多播与单播通信

  • 多播：向多播地址224.0.0.18发送VRRP包，效率高
  • 单播：直接与对端通信，穿越防火墙更好

核心参数配置

我来帮你精简易懂地总结这些参数：

global_defs（全局定义）

参数 作用
notification_email 故障时的告警邮箱
router_id 机器标识（通常用主机名）

vrrp_instance（VRRP实例）

参数 作用
state 初始状态：MASTER 或 BACKUP
interface 绑定的网卡（如 ens33、eth0）
mcast_src_ip 发送心跳的IP地址（选择稳定网卡）
virtual_router_id 组ID，相同ID为一个组
priority 优先级（1-255），高的为MASTER
advert_int 心跳间隔（默认1秒）
authentication 认证密码（主从必须一致）
virtual_ipaddress 虚拟IP地址（VIP）
track_script 引用的检测脚本名

vrrp_script（检测脚本）

参数 作用
script 检测命令或脚本
interval 检测间隔（秒）
weight 失败时降低多少优先级
fall 连续失败N次才认定为真失败
rise 连续成功N次才认定为成功

核心逻辑：脚本监控服务 → 失败时降优先级 → 优先级低的自动变BACKUP → 高优先级的变MASTER（故障转移）

总结

一、Keepalived 是什么？

Keepalived 是基于 VRRP 协议的高可用解决方案，通过虚拟IP漂移实现服务故障自动转移。

二、核心工作原理

三、三大配置模块

1. global_defs（全局定义）

  • notification_email：故障告警邮箱
  • router_id：机器标识
  • script_user：脚本运行用户

2. vrrp_instance（VRRP 实例）

  • state：初始状态（MASTER/BACKUP）
  • interface：绑定网卡
  • virtual_router_id：组ID（主备必须相同）
  • priority：优先级（主高于备）
  • virtual_ipaddress：虚拟IP（VIP）
  • authentication：认证密码（主备必须相同）
  • track_script：引用检测脚本

3. vrrp_script（检测脚本）

  • script：检测命令或脚本
  • interval：检测间隔（秒）
  • fall：连续失败N次认定故障
  • rise：连续成功N次恢复正常
  • weight：失败时优先级降幅

四、主备配置差异

配置项 主节点 备节点
state MASTER BACKUP
priority 90 85
router_id 不同 不同
virtual_router_id 相同 相同
authentication 相同 相同
virtual_ipaddress 相同 相同

五、故障转移流程

  1. 正常状态：Master 持有 VIP，每秒发送心跳包
  2. 检测失败：脚本连续失败 N 次（fall 次数）
  3. 优先级下降：Master 的 priority 值被降低
  4. VIP 漂移：Backup 优先级更高，接管 VIP
  5. 恢复正常：脚本连续成功 N 次（rise 次数），Master 状态恢复

七、实战要点

✅ 必做项

  • 关闭防火墙或放行 VRRP 多播（224.0.0.18）
  • 检测脚本必须可执行（chmod +x）
  • vrrp_instance 中必须有 track_script
  • 主备 virtual_router_id 必须相同

⚠️ 常见坑

  • 网卡名称不一致（ens33 vs eth0）
  • 密码配置不一致导致主备无法通信
  • 检测脚本逻辑错误无法正确判断故障
  • 优先级差距过小，容易频繁切换

八、测试验证

九、Keepalived + Nginx 架构价值

  • 双层高可用：Keepalived 保证入口（VIP），Nginx 保证应用
  • 透明转移：客户端无感知切换
  • 业务连续性：RTO 接近 0（秒级转移）
  • 成本低：开源方案，无额外硬件投入
