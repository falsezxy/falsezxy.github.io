MySQL 备份与恢复

引言 - 日志是什么？

谈到备份，就不得不提日志。

日志的定义与作用

日志是什么：用于记录指定对象生命周期内发生的所有事件

日志可以用来做什么：

  • ① 用于记录 - 例如：nginx access.log（访问日志）
  • ② 用于排障 - 例如：nginx error.log（错误日志）、journalctl -xe（系统日志）、journalctl -u nginx（服务日志）
  • ③ 服务日志 - 通用日志、业务日志（如Java Tomcat、Amoeba等应用日志）

业务日志补充：代码运行过程中记录的日志，部分业务日志运维可能无法直接访问，需要与开发团队协调获取

日志分类

  1. 系统日志 - 操作系统级别的日志记录
  2. 服务日志 - MySQL、Nginx等服务产生的日志
  3. ⭐⭐业务日志 - 应用程序代码运行过程中产生的日志

与MySQL备份的关系

MySQL的二进制日志（Binary Log） 就是一种服务日志，它记录所有修改数据库的操作，是增量备份和灾难恢复的关键基础。

一、备份基础概念

1.1 备份的目的

灾难恢复 - 应对数据丢失场景（程序错误、人为操作错误、磁盘故障、灾难等）

1.2 备份分类

按备份时数据库状态分类

分类 特点 工具
冷备份（脱机备份） 关闭数据库，直接打包数据文件 tar
热备份（联机备份） 数据库运行中备份 xtrabackup、mysqlhotcopy
温备份 数据库锁表（可读不可写）备份 mysqldump ⭐

按备份内容分类

  • 完全备份：整个数据库完整备份，是增量备份的基础
  • 增量备份：只备份上次完全或增量备份后的数据变化，需要逐个恢复所有增量文件
  • 差异备份：只备份上次完全备份后的所有变化，恢复时只需完全+最后一次差异

备份频次建议

  • 完全备份：5~7天一次（非业务高峰期，如晚上10点至凌晨5点）
  • 增量备份：1~3天一次
  • 差异备份：按需进行

二、完全备份与恢复

2.1 冷备份（物理备份）

原理：关闭MySQL服务，直接压缩data文件夹

优点：备份速度快，恢复简单
缺点：必须关闭数据库，业务中断

2.2 温备份（mysqldump）⭐

原理：逐表备份，每张表备份时按以下步骤：

1.对需要备份的表加读锁

  • 防止写入
  • 允许读取
  • 保证数据在一个一致时间点

2.导出表结构（SHOW CREATE TABLE）

  • 得到建表语句
  • 不会真正执行 DROP/CREATE

3.导出数据

  • 常见方法：
    • SELECT * FROM table INTO OUTFILE
    • mysqldump（内部实现也是 SELECT）
  • 只是导出，不执行 INSERT

格式 原理 优点 缺点
STATEMENT（默认） 记录SQL语句 日志量小，速度快 高并发下可能出现顺序错误、不精确

MIXED（推荐） 普通语句用STATEMENT，函数操作用ROW 兼顾性能和精确性 -

恢复流程速记

  1. 完全备份丢失 → 恢复完全备份文件 + 逐个恢复增量备份
  2. 单表数据错误 → 单独备份该表恢复
  3. 需要跳过某条SQL → 使用--stop-position断点恢复
  4. 需要回到某个时间点 → 使用--stop-datetime时间点恢复

七、总结

备份核心概念回顾

维度 内容
备份目的 灾难恢复 - 应对数据丢失
三种状态 冷备（关闭）、热备（运行）、温备（锁表）
三种策略 完全备份（全量）、增量备份（增量）、差异备份（差异）
常用工具 tar（冷备）、mysqldump（温备）、mysqlbinlog（增量）

选择备份方案的判断标准

选择冷备份还是温备份？

  • 冷备：业务可中断的情况，速度快，操作简单
  • 温备：7×24小时业务运行，数据量大，对实时性要求高

选择完全备份还是增量备份？

  • 完全备份：数据量小，每5~7天一次；恢复快，只需一个文件
  • 增量备份：数据量大，每1~3天一次；需要多个文件逐个恢复

选择STATEMENT还是ROW还是MIXED？

  • STATEMENT：日志小，但高并发下可能不精确 ✗
  • ROW：精确，但日志大，性能影响大 △
  • MIXED：推荐选择，兼顾性能和精确性 ✓

运维最佳实践

定期完全备份 + 每日增量备份 + 异地存储

恢复决策树

关键命令速查

任务 命令
备份单库 mysqldump -u root -p --databases db_name > backup.sql
备份全库 mysqldump -u root -p --all-databases > backup.sql
恢复备份 mysql -u root -p db_name < backup.sql
刷新日志 mysqladmin -u root -p flush-logs
查看日志 mysqlbinlog --no-defaults --base64-output=decode-rows -v mysql-bin.000001
位置点恢复 `mysqlbinlog –no-defaults –start-position=’X’ –stop-position=’Y’ mysql-bin.000001
时间点恢复 `mysqlbinlog –no-defaults –start-datetime=’2025-09-16 10:00:00’ –stop-datetime=’2025-09-16 11:00:00’ mysql-bin.000001

常见错误与排查

问题 原因 解决方案
mysqlbinlog 报UTF-8错误 未指定字符集 加--no-defaults参数
恢复后数据不完整 遗漏了某个增量文件 检查完全备份+增量备份的时间序列
恢复速度很慢 ROW格式日志量大 检查binlog_format配置，改为MIXED
备份文件太大 完全备份频次过高 改为57天一次完全+13天一次增量

运维清单

  • 确保binlog已启用：log-bin=mysql-bin
  • 确保binlog格式为MIXED：binlog_format=MIXED
  • 每日检查备份日志是否完整
  • 定期测试恢复流程（至少每月一次）
  • 备份文件已转移到异地存储（NAS/NFS/云存储）
  • 文档记录最近一次完全备份和增量备份的位置
  • 团队成员掌握基本的恢复操作
