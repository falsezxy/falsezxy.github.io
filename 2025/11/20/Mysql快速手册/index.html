Mysql常见知识点

mysql主从复制原理和怎么做？

主从复制——数据同步

为什么要做?

数据备份与容灾，提高可用性，业务连续性

原理：2日志3线程

将主数据库的所有操作记录在二进制日志中，然后从数据库的IO线程向主数据库的binlog dump线程建立连接，请求获取二进制日志存贮在从数据库的中继日志中，从数据库的sql线程读取中继日志执行SQL语句，以同步。

怎么做？

先同步时钟

主数据库配置my.cnf开启二进制日志、从数据库日志同步、设置server_id

grant授权从数据库权限

查看二进制日志定位

从数据库配置my.cnf开启二进制日志、中继日志、定义中继日志的index索引，设置server_id

change master to设置主数据库的ip，同步位置等配置信息

怎么做读写分离？

读写分离——在主从复制基础上，通过代理层将数据库操作进行路由：写操作转发到主数据库，读操作转发到从数据库

为什么要做？

提高并发能力，降低单点压力

怎么做？

（安装java环境，安装Amoeba）使用Amoeba作为中间件

在所有数据库中grant开放权限给Amoeba访问

代理服务器配置Amoeba.xml配置文件——读写分离，配置读是master，写是salves

代理服务器配置dbServers.xml配置文件——设置主从数据库，和登录数据库的账户

开启服务在8066端口

mysql的备份和恢复怎么做？

冷备份：关闭数据库tar

热备份：xtrabackup、mysqlhotcopy

温备份：mysqldump

备份分类

  • 完全备份：整个数据库完整备份，是增量备份的基础
  • 增量备份：只备份上次完全或增量备份后的数据变化，需要逐个恢复所有增量文件
  • 差异备份：只备份上次完全备份后的所有变化，恢复时只需完全+最后一次差异

备份频次

  • 完全备份：5~7天一次（非业务高峰期，如晚上10点至凌晨5点）
  • 增量备份：1~3天一次
  • 差异备份：按需进行

增量备份怎么做的？

配置数据库配置文件my.cnf，开启二进制日志，指定日志格式

先完全备份mysqldump

再刷新二进制日志，生成新的日志用于增量记录

mysqladmin -u root -p密码 flush-logs

MHA怎么做高可用的，怎么实现的（原理，怎么做）？

MHA（Master High Availability）——监控主库，主库故障时将从库提升为主库，保证数据库服务的连续性

为什么要做？（高可用）

冗余、容灾、备份、故障转移

避免单点故障的问题

原理？

MHA 基于 MySQL 主从复制，通过监控主库健康状态，在主库故障时自动选举一个数据最完整的从库提升为新的主库，并自动修复 binlog 差异，实现高可用切换。

怎么做？

1.首先配置主从复制，单独开设一台服务器用于mha服务，在数据库中给予权限（mha）（数据库配置过程中，每个数据库，无论主从，都要开启二进制日志的更新，用于故障转移时能切换为主库）

2.在每个服务器都安装mha服务与mha-node组件，在mha服务器安装manager组件

3.设置每个服务器的ssh认证，确保各个节点能联通（免交互）

4.配置mha，复制管理脚本(或者配置时指定)

主要配置failover脚本和app1.cnf

failover脚本用于故障漂移

app1.cnf配置manager相关内容，比如日志，目录，主从ip，登录，数据库地址池等相关配置

5.开启mastervip，启动监控

mysql双m架构（mysql+keepalived）

1.配置双m，即互为主备

2.配置keepalived实现高可用

3.设置检查脚本

mysql索引是什么？有什么作用？索引副作用？建立索引的依据？

索引——索引是 MySQL 为了加快查询速度而对表的列建立的有序数据结构，相当于一本书的目录。

作用？

索引简单概括：优化查询，提升效率

原理？

当用户使用select 语句查询表数据时，会优先遍历索引表（目录），寻找字段是否包含索引，如果有索引，则可以直接定位到数据记录行的位置，无索引，则需要遍历表来找到select数据记录行的内容

副作用？

增加磁盘压力：索引表会占用额外的磁盘空间
更新消耗：更新包含索引的表，会需要同比更新索引表的内容
索引创建的不合理：反而会降低SQL的查询效率。

创建索引的依据？

1 主键和外键：主键和外键列通常应该建立索引，因为它们经常用于查询和连接操作。
2 记录数超过300行的表：对于包含大量记录的表，应该考虑建立索引以加速查询。
3 连接字段：在经常用于连接（JOIN）操作的字段上建立索引。
4 唯一性差的字段：唯一性差的字段（如性别、布尔值等）不适合建立索引，因为索引的选择性较差。
5 更新频繁的字段：更新频繁的字段不适合建立索引，因为索引的更新开销较大。
6 WHERE子句中的字段：经常出现在WHERE子句中的字段，特别是大表的字段，应该建立索引。
7 GROUP BY和ORDER BY字段：在经常进行GROUP BY和ORDER BY操作的字段上建立索引。
8 选择性高的字段：索引应该建在选择性高的字段上，即不同值较多的字段。
9 小字段：索引应该建在小字段上，因为大字段（如长文本）的索引开销较大。

粗略的小结（主要用于回答面试题时的过渡性介绍）
一般情况：
频繁使用且唯一性较高、更新频次较低的字段，适合添加索引
唯一性弱、更新频繁的字段不要增加索引

索引类型？有什么区别？

① 主键索引 primary key 唯一且非空
② 唯一性索引 unique key 唯一，可以为空值（空值不参与比较）
③ 组合索引
④ 全文索引 full text

主键索引（PRIMARY KEY）

唯一且非空，用来唯一标识一行数据，并且是 InnoDB 的聚簇索引。

唯一索引（UNIQUE KEY）

保证字段值不重复（可为 NULL），用于约束业务唯一性。

普通索引（INDEX）

用于加速查询，没有唯一性限制。

组合索引（Composite Index）

由多个字段组成的索引，遵循最左前缀原则提高多条件查询效率。

全文索引（FULLTEXT）

用于大文本的分词搜索，实现类似搜索引擎的全文检索。

mysql的优化（调优）？

查询优化 使用 EXPLAIN 分析查询执行计划，关注 type、key、rows 等字段。添加适当索引，避免全表扫描。优化 WHERE、JOIN、GROUP BY 等子句，减少数据扫描量。

索引策略 建立单列或联合索引支持高频查询。遵循最左前缀原则。避免索引失效（如在索引列上使用函数、类型转换）。定期检查和删除冗余、未使用的索引。

表结构设计 选择合适的字段类型和长度，减小存储空间。使用 NOT NULL 和默认值，避免 NULL 导致的性能问题。适当拆分大表或使用分区。

慢查询日志 启用慢查询日志（slow_query_log），记录执行时间过长的 SQL。分析问题语句，找出瓶颈并优化。

并发与缓存 合理配置连接池和最大连接数。使用缓存（Redis 等）减少数据库访问。考虑读写分离应对高并发。

我认为上面五个主要

SQL 编写规范 避免 SELECT *，只查询需要的列。使用 LIMIT 分页而非一次加载全部数据。减少子查询层数，改用 JOIN。避免在 WHERE 条件中进行计算。

参数调优 调整 buffer_pool_size、max_connections、query_cache_size 等参数，根据硬件资源和业务需求配置。

监控与维护 定期监控 CPU、内存、磁盘 I/O 指标。执行 ANALYZE TABLE、OPTIMIZE TABLE 维护表结构。及时更新统计信息。

什么时候需要调试?

性能下降 系统响应变慢、查询耗时增加、吞吐量下降，说明现有配置或 SQL 已不适应业务量。

高 CPU/内存占用 数据库进程消耗资源异常，可能是低效查询、大量全表扫描、缓存配置不当。

慢查询日志告警 慢查询数量增加或新出现的慢查询，表明特定 SQL 或业务变化需要优化。

并发问题 连接数频繁达到上限、线程堆积、锁等待超时，说明需要提升并发能力。

磁盘 I/O 过高 读写操作频繁、磁盘利用率接近饱和，通常源于索引不足或查询不优化。

我认为上面五个重要

业务量突增 新功能上线、用户增长、数据量翻倍后系统开始出现瓶颈，需要提前调优。

频繁的锁冲突 应用日志显示 lock wait timeout、deadlock 错误，需要优化 SQL 执行时间或调整隔离级别。

定期体检 即使暂无问题，也应定期分析慢查询日志、检查索引使用效率、审视表结构，做到未雨绸缪。

可以考虑使用explain 调试SQL语句，根据显示的内容，添加索引、优化查询sql

⭐慢查询日志：主要记录的是select 语句–》记录的是执行时间过长的SQL语句
