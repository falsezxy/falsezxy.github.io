一、关系型数据库和非关系型数据库

关系型数据库（SQL）

  • 特点：
    • 表格模型（行 + 列）
    • 使用 SQL 语言
    • 数据必须符合表结构
    • 强事务 ACID
    • 纵向扩展（升级硬件）
  • 常见产品：MySQL、Oracle、PostgreSQL
  • 适用：结构化数据、一致性要求高（如银行转账）
  • 举例：银行转账：A 转账给 B，必须保证 A 扣钱成功的同时 B 收钱成功（事务保证一致性）

非关系型数据库（NoSQL）

  • 特点：
    • 键值对 / 文档 / 图结构存储
    • 无需固定表结构
    • 高并发、高可扩展
    • 横向扩展（增加服务器节点）
  • 常见产品：Redis、MongoDB、HBase、Memcached
  • 适用：非结构化数据、高并发场景（如微信聊天消息）
  • 举例：微信聊天：一条消息可能是文字、图片、语音，不适合用表格存储，更适合用文档型数据库

二、Redis简介

是什么

  • 定义：开源、C 语言编写、基于内存、支持持久化的键值数据库
  • 用途：缓存系统，为MySQL做缓冲层存储高热数据

有什么用？

  1. 给MySQL做缓冲层：存储高热数据，减轻数据库压力
  2. 存储非结构化数据：灵活存储各种数据类型
  3. 缓存系统：加速数据访问

为什么这么快？

原因 说明
纯内存操作 避免磁盘I/O，速度极快
单线程 避免锁开销，原子性强
I/O多路复用 高并发处理能力

性能指标：读取110000次/s，写入81000次/s

Redis 特性

  • 高性能：读写速度极快
  • 数据结构丰富：string、list、hash、sets、sorted sets
  • 支持持久化：数据可保存到磁盘（RDB、AOF）
  • 原子性：单线程保证操作的原子性
  • 主从复制：数据备份与高可用支持

⭐Redis持久化机制

为什么需要持久化？

Redis基于内存，停机会丢失数据。持久化的目的是断电后数据不丢失。

两种持久化方式

1. RDB（快照）

是什么：周期性把内存数据保存到硬盘的二进制文件

优点：文件小、恢复快

缺点：可能丢失几分钟的数据、对性能有一定影响

触发条件：

  • 手动触发：save（阻塞）或 bgsave（非阻塞，推荐）

  • 自动触发：配置文件中的save条件

执行流程：

  1. Redis父进程判断是否有其他子进程在执行
  2. 父进程fork创建子进程（fork时会阻塞）
  3. 子进程生成RDB文件，父进程继续处理请求
  4. 子进程完成后替换原文件
2. AOF（日志）

是什么：记录Redis执行的每条写、删命令到日志文件

优点：实时性好、数据安全、文件可读

缺点：文件大、恢复慢、对性能影响大

开启方式：

执行流程：

  1. 命令追加(append)：写命令追加到缓冲区
  2. 文件写入与同步(write/sync)：根据策略同步到硬盘
    • appendfsync always：每条命令立即同步，安全但性能差
    • appendfsync everysecond：每秒同步一次（默认推荐）
    • appendfsync no：由操作系统决定，数据不安全
  3. 文件重写(rewrite)：定期压缩AOF文件
    • 手动：bgrewriteaof
    • 自动：满足条件时触发（文件大小翻倍等）

启动加载优先级

AOF > RDB

  • AOF开启时优先加载AOF恢复数据
  • AOF关闭时才加载RDB文件
  • 当AOF开启但文件不存在时，即使RDB存在也不加载

RDB vs AOF

对比项 RDB AOF
文件大小 小 大
恢复速度 快 慢
实时性 较差 实时
性能影响 中 大
推荐场景 全量备份 主要持久化方案

五、Redis主从复制

是什么？

将一台Redis（Master）的数据复制到其他Redis服务器（Slave）。一主可多从，一从只能一主。数据复制是单向的，只能由主节点到从节点。

为什么需要主从复制？

  • 数据冗余：主从复制实现了数据的热备份
  • 故障恢复：当主节点出现问题时，可以由从节点提供服务
  • 负载均衡：主节点负责写，从节点负责读，分散读压力
  • 高可用基石：哨兵和集群都是在主从复制基础上实现的

工作流程

redis 主从复制
1、从向主发送sync请求
2、主开启bgsave进行RDB 持久化（保存文件）
3、保存后推送给从服务器，从服务器开始”恢复”数据到内存中
4、同时主将内存中新增的数据通过AOF的方式给到从服务器

RDB相当于快照，下次启动时恢复

AOF记录执行语句，启动时重新执行，以文件形式存储

六、Redis哨兵模式

是什么？

哨兵(sentinel)是一个分布式系统，在主从复制的基础上，配置哨兵程序，监控master，进行选举机制（投票）判断是否进行故障迁移。

为什么需要哨兵？

主从模式故障恢复需要人工干预，效率低。哨兵通过自动投票机制实现主节点故障的自动转移。

核心原理（工作原理）

监控→判定→选举→转移

  1. 监控：哨兵定期向主从和其他哨兵发送ping，检测存活状态
  2. 判定故障：
    • 单个哨兵判定Master不回复 → 主观下线
    • 多个哨兵同意 → 客观下线
  3. 选举新Master：通过Raft算法选举领导哨兵
  4. 故障转移：
    • 从Slave中选出最适合的升级为Master（基于优先级replica-priority、复制完整度——复制偏移量offset）
    • 其他Slave指向新Master
    • 原Master恢复后变为Slave
  • 主观下线：单个哨兵认为某节点故障
  • 客观下线：多个哨兵共同认可故障（需达到quorum值）
  • Raft算法：分布式投票选举算法

七、Redis集群模式

是什么？

集群，即Redis Cluster，是Redis 3.0开始引入的分布式存储方案。通过哈希槽分片将数据分散到多个节点。

为什么需要集群？

  • 突破单机内存限制：支持海量数据存储
  • 提高写性能：多个Master并行处理写请求
  • 自动故障转移：Master宕机自动选举Slave升级

核心原理

哈希槽(Hash Slot)：

  • Redis集群有16384个哈希槽（编号0-16383）
  • 集群的每个节点负责一部分哈希槽
  • 每个Key通过CRC16(key) % 16384计算所属槽位
  • 以3个节点为例：
    • 节点A：0-5460
    • 节点B：5461-10922
    • 节点C：10923-16383

集群的主从复制模型：

  • 为每个节点添加一个从节点，形成3主3从结构
  • Master故障时，其对应的Slave自动升级为新Master

八、总结

Redis三种架构选择

规模 架构 主要特点
小 单机 开发测试环境使用
中 主从+哨兵 企业常见，故障自动转移，建议3个哨兵
大 集群模式 海量数据、超高并发，最少6个节点

三种架构核心对比

主从复制：

  • 在主从复制的基础上配置哨兵监控master
  • 进行选举机制（投票）判断是否进行故障迁移
  • 解决：故障转移自动化
  • 缺陷：写操作无法负载均衡、存储能力受单机限制

哨兵模式：

  • 在主从复制的基础上，哨兵实现了自动化的故障恢复
  • 解决：主节点故障自动转移
  • 缺陷：写操作无法负载均衡、存储能力受单机限制

集群模式：

  • 通过哈希槽将数据分片到多个节点
  • 每个Master都可以处理读写请求
  • 解决：写操作负载均衡、存储能力突破单机限制
  • 优势：最完善的高可用方案

Redis面试核心要点

  1. 持久化：RDB快速但有数据丢失风险，AOF安全但较慢
  2. 主从：数据冗余、读写分离基础
  3. 哨兵：自动故障转移，企业推荐配置3个哨兵
  4. 集群：数据分片，突破单机限制
  5. 故障：穿透、击穿、雪崩的原理与解决方案

Redis常见问题

1.Redis 的持久化有哪些，有什么区别（怎么触发的持久化）（RDB至少两种）

RDB（Redis Database Snapshot）(快照)

AOF（Append Only File）(日志)

RDB是定时拍照保存快照，AOF是每次写操作都记录命令日志，AOF更安全但更耗性能，生产环境通常用AOF的everysecond策略做到平衡。

RDB 持久化

触发方式：

  • 手动：save（阻塞）、bgsave（后台，推荐）
  • 自动：配置文件中的save条件（如：900秒1次变化、300秒10次变化等）
  • 其他：主从复制全量复制、执行shutdown命令时

优缺点：文件小、恢复快，但可能丢失数据、对性能有影响

AOF 持久化

触发方式：

  • 自动：每次写命令执行时都追加到AOF缓冲区
  • 同步策略：always（每条立即同步）、everysecond（每秒同步）、no（操作系统决定）
  • 重写：手动bgrewriteaof或自动触发（文件大小条件）

优缺点：实时性好、数据安全，但文件大、恢复慢、性能影响大

2.Redis 哨兵模式与集群模式的工作原理分别是什么？区别是什么？哨兵/集群模式分别是怎么做的？

哨兵模式

工作原理？

在主从复制的基础上，配置哨兵程序，监控master信息，hello频道共享，故障时主客观下线，然后进行raft选举机制（投票）判断是否进行故障(master)迁移。

怎么做？

配置redis中的sentinel.conf哨兵文件，指向主库，然后启动，在出现故障时会进行投票选举判断是否进行故障迁移

特点：故障自动转移，但写操作和存储能力仍受单机限制

cluster模式

通过哈希槽将数据分散到多个节点，每个Master负责一部分槽位和读写请求

工作原理？

让master相互至少6台服务器，需要奇数数量的master，cluster集群模式进行健康检查、判断彼此存活，同时每个master都会有至少1台自己的从服务器，并且实现”主从复制”，当一个master 挂掉了，其他master也会进行选举，选择出最适合成为新master的slave服务器，同时挂掉的Master也会指定一个自己的服务器来继承slots。以保证集群slave的完整性。

怎么做？

配置redis的配置文件redis.conf，设置开启集群模式，开启持久化，设置端口，超时时间等，即可完成配置。

特点：写负载均衡、存储扩展、最完善的高可用方案

区别

哨兵只有1个Master写，集群有多个Master写；哨兵单机存储，集群多机分散存储；哨兵部署简单，集群部署复杂。

3.Redis 的优化

1、内存碎片的优化

监控碎片率，超过1.5时执行SHUTDOWN SAVE重启

原因：操作系统分配物理内存的不连续性导致

举例：Redis想存储1G数据但操作系统没有连续1G内存，只能用多个小块拼起来，多余的未直接用于数据存储的内存就是碎片

2、调整redis内存数据的⭐⭐淘汰机制和命中机制

根据场景选择合适的淘汰策略（缓存场景推荐allkeys-lru）

当Redis内存达到maxmemory限制时，触发淘汰策略：

策略 说明
noeviction 禁止删除，满时报错（严格场景）
allkeys-lru 删除最少使用的key（缓存场景推荐）
volatile-lru 仅在设置TTL的key中删除最少使用
volatile-ttl 删除剩余生存时间最短的key
allkeys-random 随机删除任意key

4.三个故障是什么？解决方案和预防方案

redis的常见故障：缓存雪崩、内存穿透、缓存击穿

1. 缓存穿透

是什么：请求查询不存在的数据，缓存无法命中，大量请求直接穿透到DB

原因：恶意攻击或程序bug频繁请求不存在的数据

解决方案：

  • 缓存空结果：把不存在的数据也存到缓存里，告诉大家”这个东西是空的”，避免重复查询DB
  • 使用布隆过滤器：在缓存前加过滤器，快速判断请求的数据是否一定不存在

预防方案：

  • 参数校验，过滤非法请求
  • 实现请求频率限制

2. 缓存击穿

是什么：某个热点数据失效时，大量针对这个数据的请求穿透到DB

原因：高频访问的热点数据突然过期

解决方案：

  • 设置热点数据不过期：对特别受欢迎的数据设置永不过期或延长过期时间
  • 加锁排队：缓存失效时只让第一个请求回源DB，其他用户等待
  • 随机退避：失效后随机sleep再次查询，如果失败再更新

预防方案：

  • 对多个热点key加固定时间+随机数，避免同时失效
  • 提前预热热点数据

3. 缓存雪崩

是什么：大量缓存数据同时失效或Redis宕机，导致大批请求涌入DB

原因：大量缓存数据在同一时间过期，或Redis故障宕机

解决方案：

  • 设置缓存过期时间的随机值：不要让所有数据在同一时间过期
  • 双缓存机制：在缓存失效前准备备份缓存
  • 加锁或限流：对DB的访问加锁或限流保护
  • 使用集群架构：通过哨兵或集群保证Redis高可用

预防方案：

  • 监控Redis运行状态
  • 使用主从+哨兵或集群架构
  • 合理规划缓存策略

ELK企业级日志分析系统
ELK企业级日志分析系统ELK是什么？ELK是一套开源日志分析系统，由三个核心组件组成： Elasticsearch：分布式搜索引擎，负责日志存储和快速检索 Logstash：日志收集处理引擎，用于收集、解析、转换日志数据 Kibana：数据可视化工具，提供友好的Web界面查询和分析日志 衍生方案：Filebeat（轻量级日志收集器，替代Logstash中的输入端） 主要目的：收集、存储、分析与可视化 为什么要用ELK？传统单机日志分析存在的问题： 日志分散在各个服务器，查询困难 海量日志无法有效存储和分析 难以快速定位和追踪问题 ELK解决方案的优势： 集中管理企业级日志，支持分布式存储 支持全文搜索，查询性能高 灵活的数据可视化和告警机制 支持海量日志实时处理 工作原理简化叙述流程： 采集阶段：Logstash/Filebeat从应用服务器、系统日志、网络设备等多种来源收集日志 处理阶段：Logstash对日志进行处理 使用Grok、JSON、正则表达式等解析器提取结构化数据 添加新字段、删除敏感信息、丰富数据内容 日志转换为标准格式 ...
