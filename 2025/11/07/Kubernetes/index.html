1.2 Kubernetes架构概述

核心模型

Kubernetes 采用客户端-服务器模型运行：

  • 开发者 通过 kubectl 命令行工具与集群交互
  • 控制平面 是集群的大脑，做出决策和管理状态
  • 工作节点 运行实际的应用程序容器

控制平面（集群的”大脑”）

控制平面由四个关键组件组成：

组件 简称 职责
kube-apiserver API 集群的前门，处理所有管理命令和资源请求
etcd 数据库 存储集群的配置数据和当前状态，丢失它意味着丢失整个集群状态
kube-scheduler 调度器 根据资源需求将 Pod 分配给合适的节点
kube-controller-manager 控制器 不断调整集群状态，确保实际状态匹配所需配置

工作节点（运行应用的机器）

每个工作节点有两个关键组件：

组件 简称 职责
kubelet 节点代理 与控制平面通信，管理节点上的 Pod 运行
Container Runtime 容器引擎 运行和管理容器（如 Docker、containerd）

核心资源

Pod（最小部署单元）

  • Kubernetes 中最小可部署单位
  • 通常包含一个容器，但可以包含多个共享网络和存储的容器
  • Pod 像是 Kubernetes 版本的”容器”

Service（服务发现与负载均衡）

  • 为一组 Pod 提供稳定的访问方式
  • 提供稳定的 IP 地址和 DNS 名称
  • 自动路由流量到 Pod，隐藏 Pod 的位置变化
  • 使应用程序可被可靠地访问

工作流程

要点

概念 类比 记住什么
控制平面 公司管理部门 做决策，管理全局
工作节点 工厂车间 执行决策，做实际工作
Pod 包装的产品 最小工作单元，可能包含多个容器
Service 前台接待处 提供统一入口，隐藏内部变化
kubelet 车间主任 监督节点上的工作进行
etcd 档案室 保存所有重要数据

为什么这样设计？

✓ 解耦：控制平面与工作节点分离，便于扩展
✓ 高可用：多个组件协同工作，单点故障不影响整体
✓ 自愈能力：kubelet 和控制器不断监控和调整状态
✓ 灵活扩展：添加新节点来扩展容量
✓ 声明式管理：只需声明所需状态，Kubernetes 自动实现

1.4 Kubernetes 命名空间与资源隔离

什么是命名空间？

命名空间是 Kubernetes 集群内的逻辑分区，用于组织和隔离资源。它像是一个虚拟集群，允许多个用户、团队或应用共享同一物理集群。

命名空间的优势

优势 说明
改进组织 按项目、团队或环境（开发、测试、生产）分组资源
增强安全性 限制用户和服务账户对特定命名空间的访问权限
简化管理 在特定命名空间应用资源限制、网络策略等配置
灾难隔离 一个命名空间的问题不影响其他命名空间

常见命名空间

命名空间 用途 包含内容
kube-system 系统组件 CoreDNS、kube-apiserver、etcd、调度器等
default 默认工作负载 用户创建的应用和服务（无指定命名空间时）
kube-public 公开数据 集群级别信息（所有用户可读）
自定义 业务隔离 用户为不同团队/项目创建的命名空间

2.2 基础的kubectl命令和语法

什么是 kubectl？

kubectl 是 Kubernetes 的命令行工具，就像遥控器一样，让你能远程控制和管理整个 Kubernetes 集群。

命令语法结构

所有 kubectl 命令都遵循这个模式：

  • kubectl：命令行工具名
  • [command]：你要执行的操作（如 get、describe、create 等）
  • [TYPE]：资源类型（如 pods、deployments、services 等）
  • [NAME]：具体资源名称（可选，省略则作用于该类型的所有资源）
  • [flags]：可选参数，用来修改命令行为（如 -n namespace、-o wide）

核心命令详解

命令 用途
get 查看资源列表
describe 查看资源详细信息
create 创建新资源
delete 删除资源
apply 应用/更新配置（最常用）

常见资源类型

  • pods：Kubernetes 最小的运行单位，一个 Pod 通常包含一个容器
  • deployments：用来管理多个 Pod 副本，支持滚动更新和扩缩容
  • services：暴露 Pod 中的应用程序，提供访问入口
  • nodes：集群中的工作节点（物理机或虚拟机）
  • namespaces：命名空间，用来逻辑隔离资源

2.3 创建YAML清单

Kubernetes 对象基础

在写 YAML 清单之前，先认识三个核心对象，它们是 Kubernetes 应用管理的基石。

Pod（豆荚）

Pod 是 Kubernetes 最小的运行单位，就像一个容器的”包装盒”。一个 Pod 可以包含一个或多个容器，这些容器共享网络和存储空间。简单来说，一个 Pod ≈ 你应用的一个实例。

Deployment（部署）

部署用来管理多个 Pod。它的核心职责是：确保任何时刻都有指定数量的 Pod 副本在运行。如果某个 Pod 故障了，部署会自动创建新的 Pod 替换它。部署还支持滚动更新，让你能安全地更新应用。

Service（服务）

由于 Pod 会被频繁创建和销毁，它们的 IP 地址会不断变化。Service 的作用就是提供一个稳定的访问入口——给 Pod 一个固定的 IP 和 DNS 名称，这样外部用户或其他应用就能可靠地访问你的应用，无需关心 Pod 的变化。

它们的关系： Deployment 管理 Pod，Service 暴露 Pod 中的应用，外部客户通过 Service 访问。

什么是 YAML 清单？

YAML 清单就是一个用 YAML 格式写的配置文件，用来描述你想在 Kubernetes 中创建或管理什么资源。

使用 YAML 的三大优势：

  1. 声明式管理 - 你只需要描述”想要什么状态”（比如”我要 3 个 Pod 副本”），Kubernetes 会自动调整实际状态来匹配你的期望。这比手动命令更简洁高效。
  2. 版本控制 - YAML 文件是纯文本，可以用 Git 等工具管理。这让你能追踪每次配置变化、回滚到之前的版本、与团队协作。
  3. 可重用可移植 - 同一份 YAML 清单可以在不同环境（开发、测试、生产）中使用，只需稍作修改。部署变得更一致、更可靠。

核心建议

✅ 始终使用 kubectl apply 来管理 Kubernetes 资源，原因包括：

  • 支持增量更新（只改变有差异的部分）
  • 易于跟踪配置历史

  • 更灵活，更适合长期维护

这种声明式的方式就像告诉 Kubernetes”我想要这样的状态”，而不是”给我执行这个命令”，更符合基础设施即代码（IaC）的理念。

Service的三大核心作用

1. 服务发现（Service Discovery）

在Kubernetes中，Pod是临时的，会不断创建和销毁。Pod的IP是动态变化的，你无法提前知道Pod的IP地址。

Service解决这个问题：它提供一个稳定的DNS名称和虚拟IP，无论Pod怎么变化，Service的名称和IP始终保持不变。

2. 负载均衡（Load Balancing）

Service会自动将流量分散到后面的多个Pod副本，实现负载均衡。

这样即使某个Pod负载很高，其他Pod也能分担流量。

3. 对外暴露应用（Exposing Applications）

通过不同的Service类型，可以选择如何暴露应用：

  • ClusterIP（默认）— 只在集群内部可访问
  • NodePort — 通过Node的IP:端口暴露到集群外
  • LoadBalancer — 通过云厂商的负载均衡器暴露

没有Service会怎样？

想象一下没有Service的场景：

问题来了：

  • ❌ Pod被重启了，IP变成了 10.1.1.8，前端请求失败
  • ❌ 添加新Pod时，前端代码没法利用
  • ❌ Pod扩容/缩容时，前端不知道该访问哪个Pod
  • ❌ 无法自动负载均衡

有了Service就解决了所有这些问题：

  • ✅ Pod重启，IP变化，但Service名称不变
  • ✅ 自动发现新的Pod
  • ✅ 自动负载均衡
  • ✅ 前端代码无需改动

类比理解

可以把Service比作电话簿：

  • Pod = 手机（具体的设备）
    • 经常更换、损坏、重启
    • IP地址（手机号）是动态的
  • Service = 电话簿里的联系人名字（比如”客服部”）
    • 相对稳定
    • 即使更换了手机号，电话簿里的名字不变
    • 拨打”客服部”自动转接到对应的手机

前端应用就像拨号者，不需要记住每个Pod的IP（手机号），只需记住Service的名称（联系人名字），剩下的转接工作由Service（电话簿系统）自动处理。

总结

Service的作用 = 解耦前后端 + 自动发现 + 负载均衡 + 稳定暴露

它是Kubernetes中最重要的网络抽象，没有Service，微服务之间的通信就会变得非常脆弱。

方式 用法 哲学
声明式 kubectl apply -f file.yaml 告诉 K8s “我要什么状态”
