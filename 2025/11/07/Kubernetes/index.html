1.2 Kubernetes架构概述

核心模型

Kubernetes 采用客户端-服务器模型运行：

  • 开发者 通过 kubectl 命令行工具与集群交互
  • 控制平面 是集群的大脑，做出决策和管理状态
  • 工作节点 运行实际的应用程序容器

控制平面（集群的”大脑”）

控制平面由四个关键组件组成：

组件 简称 职责
kube-apiserver API 集群的前门，处理所有管理命令和资源请求
etcd 数据库 存储集群的配置数据和当前状态，丢失它意味着丢失整个集群状态
kube-scheduler 调度器 根据资源需求将 Pod 分配给合适的节点
kube-controller-manager 控制器 不断调整集群状态，确保实际状态匹配所需配置

工作节点（运行应用的机器）

每个工作节点有两个关键组件：

组件 简称 职责
kubelet 节点代理 与控制平面通信，管理节点上的 Pod 运行
Container Runtime 容器引擎 运行和管理容器（如 Docker、containerd）

核心资源

Pod（最小部署单元）

  • Kubernetes 中最小可部署单位
  • 通常包含一个容器，但可以包含多个共享网络和存储的容器
  • Pod 像是 Kubernetes 版本的”容器”

Service（服务发现与负载均衡）

  • 为一组 Pod 提供稳定的访问方式
  • 提供稳定的 IP 地址和 DNS 名称
  • 自动路由流量到 Pod，隐藏 Pod 的位置变化
  • 使应用程序可被可靠地访问

工作流程

要点

概念 类比 记住什么
控制平面 公司管理部门 做决策，管理全局
工作节点 工厂车间 执行决策，做实际工作
Pod 包装的产品 最小工作单元，可能包含多个容器
Service 前台接待处 提供统一入口，隐藏内部变化
kubelet 车间主任 监督节点上的工作进行
etcd 档案室 保存所有重要数据

为什么这样设计？

✓ 解耦：控制平面与工作节点分离，便于扩展
✓ 高可用：多个组件协同工作，单点故障不影响整体
✓ 自愈能力：kubelet 和控制器不断监控和调整状态
✓ 灵活扩展：添加新节点来扩展容量
✓ 声明式管理：只需声明所需状态，Kubernetes 自动实现

1.4 Kubernetes 命名空间与资源隔离

什么是命名空间？

命名空间是 Kubernetes 集群内的逻辑分区，用于组织和隔离资源。它像是一个虚拟集群，允许多个用户、团队或应用共享同一物理集群。

命名空间的优势

优势 说明
改进组织 按项目、团队或环境（开发、测试、生产）分组资源
增强安全性 限制用户和服务账户对特定命名空间的访问权限
简化管理 在特定命名空间应用资源限制、网络策略等配置
灾难隔离 一个命名空间的问题不影响其他命名空间

常见命名空间

命名空间 用途 包含内容
kube-system 系统组件 CoreDNS、kube-apiserver、etcd、调度器等
default 默认工作负载 用户创建的应用和服务（无指定命名空间时）
kube-public 公开数据 集群级别信息（所有用户可读）
自定义 业务隔离 用户为不同团队/项目创建的命名空间

要点

✓ 命名空间不是隔离：不同命名空间的 Pod 可以通过 DNS 互相通信
✓ 默认行为：不指定命名空间时，资源创建在 default 命名空间
✓ 系统组件隔离：kube-system 中的组件由 Kubernetes 自动管理
✓ 多环境分离：用不同命名空间表示开发、测试、生产环境
✓ 访问控制：可通过 RBAC 限制用户对特定命名空间的权限

关键要点总结

  • 命名空间是 Kubernetes 中的逻辑隔离单位
  • kube-system 包含系统组件，default 包含用户应用
  • 使用 -A 或 --all-namespaces 查看整个集群资源
  • 可创建自定义命名空间实现应用隔离和组织
  • 命名空间便于多团队共享一个集群，同时保持资源独立
